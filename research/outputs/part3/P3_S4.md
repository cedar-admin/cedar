## Section 1: Complete Source Type → Authority Level Mapping

### Methodology Notes

Authority level represents the legal weight of a regulatory entity — its binding force and standing in the hierarchy of legal authorities. The mapping below follows these foundational legal principles:

- **Statutes** are enacted by legislatures and have the highest authority within their jurisdiction; regulations must be authorized by statute.
- **Regulations** (rules in the Code of Federal Regulations or state administrative codes) are promulgated by agencies under statutory delegation; they carry full binding force equal to statute for parties subject to them.
- **Sub-regulatory guidance** (agency guidance documents, policy statements, FAQs, letters) lacks binding legal force; it represents the agency's interpretation but cannot create new legal obligations.
- **Coverage determinations** (NCDs/LCDs) operate on a Medicare-specific parallel track; they determine what Medicare will pay for, not what is lawful — practitioners can provide non-covered services but cannot bill Medicare for them.
- **Professional standards** are non-binding on regulators but function as the standard of care in malpractice litigation and can be incorporated by reference into regulations (which elevates them to regulatory authority only for that incorporation).

| Source Type | Document Subtype | Authority Level | Reasoning and Exceptions |
|---|---|---|---|
| AMA policy statements | H-policy, D-policy, AMA House of Delegates resolutions | `professional_standard` | AMA policy represents the organized medical profession's consensus positions. AMA policies have no regulatory force but inform the standard of care and influence legislative and regulatory positions. `professional_standard` is the correct classification. |
| AMA | CPT code editorial guidance (not the CPT code set itself) | `professional_standard` | CPT editorial guidance explaining code usage is AMA professional guidance. They do not have regulatory force in themselves. **Important exception:** If a Florida statute or F.A.C. rule incorporates a specific guideline by reference (e.g., "practitioners must follow the guidelines of [Society]"), that regulatory reference elevates the guideline's practical authority, but Cedar still classifies the guideline document itself as `professional_standard` and the regulatory provision incorporating it as `state_board_rule`. |

| Accreditation standards | The Joint Commission (JCAHO) — hospital accreditation standards | `professional_standard` | **Nuanced case:** TJC accreditation standards are not regulations. The regulatory obligation is in 42 CFR; TJC standards are the mechanism for demonstrating compliance. |

## Section 2: Within-Source Authority Level Disambiguation

### 2A. Federal Register — The Primary Disambiguation Challenge

The Federal Register publishes content at four distinct authority levels through the same channel. Deterministic disambiguation is achievable for the majority of cases.

#### Metadata Fields for FR Disambiguation

The FR API exposes the following fields that drive disambiguation:

| FR API Field | Values/Format | Use in Disambiguation |
|---|---|---|
| `type` | `RULE`, `PRORULE`, `NOTICE`, `PRESDOCU` | Primary classifier — maps directly to base authority levels |
| `action` | Free-text string (e.g., "Final rule.", "Proposed rule.", "Notice of proposed rulemaking.") | Secondary confirmation; resolves ambiguity within `RULE` type (final vs. interim final) |
| `agencies` | Array of agency objects with `slug` | Identifies issuing agency for exception routing (DEA scheduling notices, CMS NCD notices) |
| `presidential_document_type` | `EXECORD`, `PRMEMO`, `PROCLA`, `ADMORDER` | Disambiguates within `PRESDOCU` type |
| `significant` | Boolean | Not an authority-level signal, but indicates significant rules (§553 APA criteria) |
| `cfr_references` | Array of CFR citations | Enables cross-referencing with eCFR structural rules for domain assignment |
| `subjects` | Array of subject terms | Supplementary keyword signals |
| `abstract` | Free text | Fallback for edge cases; AI disambiguation if needed |

#### FR Document Type → Authority Level Mapping (Deterministic Rules)

```
type = 'RULE'
  → default: federal_regulation
  → if action contains 'interim final': federal_regulation (note interim status in metadata)
  → if action contains 'direct final': federal_regulation (note direct final in metadata)

type = 'PRORULE'
  → default: federal_regulation + status = 'proposed'
  → if action contains 'ANPRM' or 'advance notice': federal_regulation + status = 'proposed' + metadata.stage = 'anprm'
  → if action contains 'supplemental notice': federal_regulation + status = 'proposed' + metadata.stage = 'snprm'

type = 'NOTICE'
  → Exception check 1 (run first): agencies.slug contains 'drug-enforcement-administration'
      AND (action contains 'schedule' OR subjects contains scheduling term)
      → federal_regulation (DEA scheduling via notice procedure)
  → Exception check 2: agencies.slug contains 'centers-for-medicare-and-medicaid-services'
      AND (action contains 'national coverage determination' OR title contains 'NCD')
      → national_coverage_determination
  → Default: sub_regulatory_guidance

type = 'PRESDOCU'
  → presidential_document_type = 'EXECORD': federal_statute
  → presidential_document_type = 'PROCLA'
      AND (action contains 'emergency' OR subjects contain 'public health emergency' OR 'national emergency')
      → federal_statute
  → presidential_document_type = 'PROCLA' (other): sub_regulatory_guidance
  → presidential_document_type = 'PRMEMO': sub_regulatory_guidance
  → presidential_document_type = 'ADMORDER': sub_regulatory_guidance
  → presidential_document_type = null or other: sub_regulatory_guidance
```

**Coverage: deterministic for ~97% of FR content.** The 3% requiring AI disambiguation are notices where the action field is ambiguous or missing. Cedar should route these to Stage 4 (AI classification) with the partial classification `NOTICE → sub_regulatory_guidance (default)` and the review flag set.

#### DEA Scheduling Notice Detection Logic

DEA uses the Notice procedure for scheduling actions because the Controlled Substances Act grants DEA scheduling authority outside normal notice-and-comment rulemaking. Detection is deterministic:

```typescript
function isSchedulingNotice(entity: FederalRegisterEntity): boolean {
  const deaSlugs = ['drug-enforcement-administration'];
  const hasDeaAgency = entity.agencies?.some(a => deaSlugs.includes(a.slug));

  const schedulingTerms = [
    'schedule', 'scheduling', 'controlled substance',
    'temporary placement', 'permanent placement',
    'schedule i', 'schedule ii', 'schedule iii', 'schedule iv', 'schedule v'
  ];

  const actionLower = (entity.action || '').toLowerCase();
  const subjectsLower = (entity.subjects || []).map(s => s.toLowerCase());
  const titleLower = (entity.title || '').toLowerCase();

  const hasSchedulingSignal = schedulingTerms.some(term =>
    actionLower.includes(term) ||
    subjectsLower.some(s => s.includes(term)) ||
    titleLower.includes(term)
  );

  return hasDeaAgency && hasSchedulingSignal;
}
```

### 2B. State Sources — Board Decision Disambiguation

Florida board meeting minutes and orders require disambiguation between binding adjudicative actions and non-binding deliberations.

#### Binding vs. Non-Binding Signals

| Signal | Binding (`state_board_rule`) | Non-Binding (`sub_regulatory_guidance`) |
|---|---|---|
| Document title | "Final Order," "Consent Agreement," "Emergency Order," "Declaratory Statement" | "Meeting Minutes," "Workshop Minutes," "Draft Policy," "Presentation" |
| Case number | Present (e.g., DOH-24-1234-MQA) | Absent |
| Parties named | Respondent(s) named | No specific respondent |
| Disposition | License disciplinary action (suspension, revocation, probation, reprimand, fine) | Discussion or vote on policy position |
| Signature | Board chair signature + agency clerk attestation | Chair signature only or unsigned |
| DOAH reference | Division of Administrative Hearings case number present | Absent |
| F.S. §120 citation | References to APA procedures | Absent |

**Disambiguation approach: deterministic with AI fallback.**

For web-scraped board documents, Cedar applies a header/title pattern match first:
- Pattern matches "Final Order", "Consent Agreement", "Emergency Order" → `state_board_rule`, status = 'active'
- Pattern matches "Proposed Rule", "Proposed Amendment" → `state_board_rule`, status = 'proposed'
- Pattern matches "Meeting Minutes", "Workshop" → `sub_regulatory_guidance`
- No pattern match and document < 5 pages → route to AI disambiguation
- No pattern match and document > 5 pages (likely a formal order) → route to AI disambiguation with default `state_board_rule`

### 2C. CMS Transmittals — Disambiguation

CMS transmittals update the Internet-Only Manuals. The transmittal ID and implementing regulation field help determine whether the transmittal is implementing a regulation (still `sub_regulatory_guidance` — the transmittal is sub-regulatory; the regulation it implements is separate) or correcting a prior transmittal (also `sub_regulatory_guidance`). All transmittals are `sub_regulatory_guidance` — no exceptions.

### 2D. openFDA Adverse Event Data — Special Handling

Adverse event reports (FAERS/MAUDE) are data records, not regulatory documents. Cedar should:
1. Assign `sub_regulatory_guidance` as the authority level
2. Set `metadata.document_category = 'data_record'` to distinguish from interpretive guidance
3. Suppress these from the regulation library display — they are relevant for monitoring (noting a surge in adverse events for a drug) but are not authoritative text
4. Apply a separate ingestion pathway that aggregates signals rather than treating each report as an individual regulatory entity

## Section 3: Authority Level Hierarchy and Conflict Resolution

### 3A. Explicit Hierarchy Ordering

The authority level enum maps to a numeric hierarchy rank for comparison queries and conflict detection. The scale uses non-contiguous integers to allow future insertions.

```
Rank 100: federal_statute
Rank 90:  federal_regulation
Rank 80:  sub_regulatory_guidance
Rank 70:  national_coverage_determination  (parallel track — not ranked above/below federal_regulation)
Rank 60:  local_coverage_determination     (parallel track — ranked below NCD)
Rank 50:  state_statute
Rank 40:  state_board_rule
Rank 20:  professional_standard
```

**Notes on the parallel tracks:**

NCDs and LCDs do not fit neatly into the federal hierarchy because they regulate Medicare coverage decisions, not legal conduct. A federal regulation could preempt state law; an NCD cannot preempt state law — it only controls what Medicare will pay for. The rank values assigned (70/60) place them between sub_regulatory_guidance and state_statute for sorting purposes, but Cedar's conflict resolution logic treats them as operating on a separate track (see 3B).

### 3B. Federal Preemption vs. More-Restrictive State Rules

#### The Core Legal Principle

Federal law preempts conflicting state law under the Supremacy Clause (U.S. Const. Art. VI). However, states can enact laws that are MORE restrictive than federal minimums without conflict — they are supplementing federal law, not contradicting it. This creates three relationship types:

| Relationship | Description | Example |
|---|---|---|
| Federal preemption | Federal law explicitly or implicitly displaces state law | Federal CSA scheduling preempts state attempt to legalize Schedule I controlled substances |
| State exceeds federal minimum | State law imposes additional requirements beyond federal floor | Florida requiring more frequent CME than federal guidelines |
| Independent state authority | State regulates areas where federal law is silent | Florida corporate practice of medicine prohibition (no federal equivalent) |

#### Cedar's Conflict Resolution Display Logic

Cedar cannot automatically determine the legal relationship between federal and state entities without AI analysis. The following rules define what Cedar surfaces:

**Rule 1: Same-topic detection.** When two entities in Cedar's corpus are tagged to the same L2 subdomain (from P1_S4 taxonomy) AND one has a federal authority level AND the other has a state authority level, Cedar flags them as potentially related.

**Rule 2: Conflict relationship tagging.** Cedar does NOT automatically determine whether a state rule conflicts with, supplements, or is independent of a federal rule. This determination requires legal analysis. Instead:
- Cedar surfaces both entities together in the regulation library when they share the same L2 subdomain
- Cedar displays a "See also: federal requirements" / "See also: Florida requirements" cross-reference
- Cedar adds an AI-generated note (Subject to disclaimer) on the change alert when a new state rule touches a domain with active federal regulation: "Note: Federal regulations also govern this area. This state rule may supplement or exceed federal requirements. Consult counsel to determine which requirements apply to your practice."

**Rule 3: More-restrictive determination.** Cedar does NOT make a deterministic "more restrictive" determination. This is a legal judgment requiring analysis of scope, penalties, and exemptions. Cedar's AI summary for changes in domains with both federal and state coverage will include: "Compared to federal requirements, this [Florida rule / Florida statute] [is more restrictive in the following respects / addresses areas not covered by federal law / aligns with federal requirements] — [AI analysis]." The disclaimer that this is not legal advice applies.

**Rule 4: NCD/LCD parallel track.** NCDs and LCDs are never in "conflict" with federal regulations or state laws in the preemption sense — they govern Medicare reimbursement. Cedar surfaces them alongside relevant regulations under the same domain but does not apply conflict resolution logic between the coverage determination track and the regulatory track.

**Rule 5: Professional standards.** Professional standards never supersede any legal authority. Cedar does not apply conflict resolution logic to `professional_standard` entities. They are displayed as informational context, not as competing authority.

## Section 4: Proposed Entity Protocol

### 4A. Recommendation: Status Modifier, Not Enum Expansion

**Recommendation: Use a `status` column with a `proposed` value, not enum expansion.**

**Rejected approach:** Adding `proposed_federal_regulation`, `pending_state_statute`, etc. to the authority_level enum.

**Reasoning for rejection:**

1. **Query complexity.** Every query filtering on authority level would need to include both the base level and the proposed variant: `authority_level IN ('federal_regulation', 'proposed_federal_regulation')`. This doubles the cardinality of every filter predicate and introduces an ongoing maintenance burden as new authority levels are added.

2. **Semantic mismatch.** A proposed federal regulation IS a federal regulation in the process of becoming final. Its authority level category is correct; its lifecycle status is different. These are orthogonal dimensions. Conflating them in a single enum mixes classification with status.

3. **Hierarchy ordering.** The hierarchy rank table works correctly with a separate status field. With enum expansion, the hierarchy table would need entries for every proposed variant — doubling its size with redundant rank values.

4. **UI rendering.** The UI needs to display the base authority level (what it will become) AND the proposed status. With enum expansion, the UI must parse the enum value to extract both pieces. With a status field, both are directly available.

5. **Transition tracking.** When a proposed rule becomes final, updating `status = 'active'` is a single column update (within Cedar's allowed update columns). If the authority level itself changed, it would imply the entity was classified incorrectly from the start.

**Implemented approach:** The `changes` table (or corpus entity table) includes:

```sql
authority_level authority_level NOT NULL,  -- The base authority level
status text NOT NULL DEFAULT 'active'       -- 'active' | 'proposed' | 'superseded' | 'expired' | 'pending_enforcement'
```

**Status values:**
- `active` — currently in force
- `proposed` — proposed rule/bill/guidance under review, no current legal force
- `pending_enforcement` — final rule published, effective date in the future
- `superseded` — replaced by a newer version (entity retained for audit trail)
- `expired` — emergency rule or temporary provision that has lapsed

### 4B. Authority Level Transitions

**When a proposed rule becomes final:**

The transition is a significant regulatory event and should generate a Cedar change alert.

**Entity lifecycle:**

```
INGEST: Proposed rule (status = 'proposed')
  ↓
MONITOR: Change detection on the source URL detects final rule publication
  ↓
CREATE: New entity OR update existing entity's status field

```

**Decision: Create a new entity, do not mutate the proposed entity.**

**Reasoning:**
- Cedar's `changes` table is append-only for audit reasons. A proposed rule and a final rule are distinct regulatory events. The proposed rule entity represents the period of public comment; the final rule entity represents the binding obligation.
- The proposed entity is retained with `status = 'superseded'` and a `superseded_by` reference to the final rule entity.
- This creates a complete audit trail: practitioners can see when they were alerted to the proposal and when they were alerted to finalization.

**Transition workflow:**

```
1. Change detection identifies new final rule publication
2. Intelligence pipeline classifies it as federal_regulation (final)
3. System searches for prior proposed entity with matching:
   - Same source (FR document number or docket number)
   - authority_level = 'federal_regulation'
   - status = 'proposed'
4. If match found:
   a. New entity created: authority_level = 'federal_regulation', status = 'active'
      (or 'pending_enforcement' if effective date is in the future)
   b. Prior proposed entity: status updated to 'superseded', superseded_by = new entity id
   c. Alert generated: "This proposed rule is now final. [Summary of final rule vs. proposed rule changes]"
5. If no match found:
   a. New entity created normally
   b. Alert generated: "New final rule published. [Summary]"
```

**Transition audit trail entries:**
- Change record for proposed rule detection (alert: "Proposed rule published — comment period open")
- Change record for finalization (alert: "This proposed rule is now final — effective [date]")
- Both records linked via the docket/document cross-reference in metadata

**For Florida bills:**
- Same lifecycle: proposed → enacted
- Match on bill number (e.g., HB 1234 / SB 567)
- When signed: new entity with `state_statute`, status = `pending_enforcement` if not yet in effect, or `active` if immediately effective
- Prior bill entity: status = `superseded`

### 4C. Effective Date Handling

**Scenario:** A federal regulation is published today (March 23, 2026) with an effective date of June 21, 2026 (90-day delayed effective date). The rule is final and binding-in-waiting.

**Cedar's treatment:**

1. **Authority level:** `federal_regulation` — the rule IS a final regulation; its effective date does not change its classification.

2. **Status:** `pending_enforcement` — this dedicated status value communicates that the rule is final but not yet enforceable. It differs from `proposed` (which has no legal force at all) and from `active` (which is currently enforceable).

3. **Effective date field:** Entity metadata includes `effective_date` (ISO date). Cedar's change feed displays "Effective [June 21, 2026]" prominently.

4. **Compliance countdown:** Cedar's UI shows a countdown or date display in the change detail: "Becomes enforceable in X days." This is a display feature, not a data field — computed from `effective_date` vs. current date.

5. **Status transition at effective date:** Cedar's scheduler runs a daily job checking for entities where `status = 'pending_enforcement'` AND `effective_date <= today`. On the effective date, status is updated to `active` and a notification is generated: "Regulation now in effect as of [date]." This is a system-generated change event, not a new FR publication.

6. **Interim compliance planning:** Cedar's AI summary for `pending_enforcement` entities includes a note: "This rule was finalized on [publication date] and becomes enforceable on [effective date]. Practices have [N] days to come into compliance." This directly addresses the stated Cedar value proposition of giving practices time to respond.

**Summary of status values and their meaning:**

| Status | Legal Force | User Message |
|---|---|---|
| `proposed` | None | "Under consideration — no compliance obligation yet" |
| `pending_enforcement` | Final, not yet enforceable | "Final rule — compliance required by [effective date]" |
| `active` | Fully enforceable | "Currently in effect" |
| `superseded` | Replaced | "Replaced by [newer entity]" |
| `expired` | Lapsed | "This emergency rule has expired" |

## Appendix A: Edge Cases and Unresolved Classifications

### A1. State Analog to Sub-Regulatory Guidance

Florida does not have a direct analog to federal sub-regulatory guidance because Florida's Administrative Procedure Act (Chapter 120) is stricter about distinguishing rules (adopted under Chapter 120) from non-rule guidance. Florida has an "unadopted rule" doctrine that renders invalid any agency statement that meets the definition of a "rule" but was not adopted through rulemaking. This means Florida agencies are less likely to issue binding guidance outside the rulemaking process.

**Implication for Cedar:** The `sub_regulatory_guidance` level is appropriate for FL DOH guidance, but Cedar should flag entities tagged `sub_regulatory_guidance` for the Florida jurisdiction in the UI as: "This guidance has not been adopted as a rule. Florida law limits agency reliance on non-rule guidance."

### A2. USP Standards — Hybrid Case

USP standards occupy a hybrid position because some CFR sections require compliance with USP standards as incorporated references. Cedar's classification rule:
- The USP document itself: `professional_standard`
- The CFR section incorporating the USP standard: `federal_regulation`
- The relationship: tracked in entity metadata via `incorporates_by_reference` field
- Display: the `federal_regulation` entity shows "Incorporates USP [chapter] by reference" with a link to the USP entity

### A3. FDA Warning Letters

FDA warning letters are not in the source type table above because they are enforcement correspondence, not published regulatory documents. If Cedar ingests warning letters (e.g., from FDA's Warning Letters database), they should be classified:
- Authority level: `sub_regulatory_guidance` with `metadata.document_category = 'enforcement_correspondence'`
- These are neither regulations nor formal enforcement actions — they are FDA's notification that it believes a violation exists, giving the recipient opportunity to correct

### A4. Interstate Compact Rules

The Interstate Medical Licensure Compact (IMLC) and similar compacts create interstate obligations that are authorized by state statutes but operate across multiple states. Cedar's approach:
- The compact statute enacted by Florida: `state_statute` (FL jurisdiction)
- Compact commission rules/policies: `state_board_rule` (FL jurisdiction, noting multi-state scope in metadata)
- This is a simplification; compact law is complex. Cedar should note in the AI summary when an entity involves interstate compact obligations.

### A5. Social Security Act Provisions

The Social Security Act (SSA) provides the statutory authority for Medicare and Medicaid. SSA provisions referenced in CMS context are `federal_statute`. However, Cedar is unlikely to ingest the SSA directly (it's stable legislation); what Cedar ingests is CMS regulations implementing SSA provisions (21 CFR Title 42 → `federal_regulation`) and CMS guidance interpreting them (`sub_regulatory_guidance`). SSA provisions should be referenced in entity metadata as the statutory basis but are not typically ingested as standalone entities.

## Appendix B: Integration Points with P2_S1 Pipeline

Authority level assignment runs in parallel with domain classification in the pipeline, not sequentially. Both outputs are produced from the same entity at the same pipeline stage and written to the classification result together.

**Where authority level assignment happens:**

- **Stage 1 (Structural):** If a CFR citation match resolves domain classification, the source_type (`ecfr`) also resolves authority level deterministically → write `federal_regulation` immediately.
- **Stage 2 (Metadata):** Document_type is available at this stage. FR type codes, CMS document types, and FL source types all resolve authority level deterministically here. This is the primary authority level assignment stage.
- **openFDA branch:** Source type `openfda` + dataset endpoint → authority level is deterministic per the mapping table → write during the openFDA branch classification.
- **Stage 4 (AI):** The ~3% of FR notices that require AI disambiguation get authority level resolved here alongside domain disambiguation.

**Data written to classification result:**

```typescript
interface ClassificationResult {
  // ... existing fields from P2_S1 ...
  authority_level: AuthorityLevel;       // The assigned authority level enum value
  entity_status: EntityStatus;           // 'active' | 'proposed' | 'pending_enforcement' | ...
  effective_date?: string;               // ISO date if applicable
  authority_level_confidence: number;    // 1.0 for deterministic; <1.0 for AI-assisted
  authority_level_rule_id?: string;      // FK to classification_rules.id for traceability
}
```

**No pipeline redesign required.** The `classification_rules` table already supports the `rule_type = 'authority_level'` extension. The Inngest orchestrator stages remain unchanged; authority level assignment is an additional output written alongside domain classification at each stage where it can be resolved.
