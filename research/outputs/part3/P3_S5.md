## Section 1: Citation Format Abstraction Layer

### 1A. NormalizedCitation TypeScript Interface

Every state parser must produce a `NormalizedCitation` — a common envelope the pipeline can consume regardless of source jurisdiction. The interface extends P2-S1's `ParsedCitation` without breaking it; the generic fields are sufficient for federal routing, and the state-specific fields fill in when a state parser runs.

```typescript
/**
 * NormalizedCitation — output of any StateCitationParser implementation.
 *
 * The pipeline uses this object for three purposes:
 *   1. Routing: jurisdiction + structural_key → load correct RuleSet
 *   2. Domain mapping: structural_key → chapter-level domain lookup
 *   3. Display: formatted_citation rendered in alert summaries and source links
 *
 * Extends P2-S1's ParsedCitation. Fields prefixed with _ext are state-specific
 * and should be accessed only by state-specific processors, never by the
 * generic pipeline stages.
 */
interface NormalizedCitation {
  // ── Routing fields (required by pipeline) ──────────────────────────────────

  /** ISO 3166-2 subdivision code or 'federal'. Examples: 'FL', 'CA', 'TX', 'NY' */
  jurisdiction: string;

  /**
   * The canonical code system identifier. Determines which parser produced
   * this citation and how to look up rules.
   * Examples: 'FAC', 'TAC', 'CCR', 'NYCRR', 'CFR'
   */
  code_system: CitationCodeSystem;

  /**
   * Top-level numeric title. Directly usable as a lookup key in structural rules.
   * - FAC:   64 (DOH), 59 (AHCA)
   * - TAC:   22 (Examining Boards), 25 (Health Services)
   * - CCR:   16 (Professional and Vocational Regulations)
   * - NYCRR: 10 (Health)
   * - CFR:   21 (FDA/DEA), 42 (CMS), 45 (OCR/HIPAA)
   */
  title: number;

  /**
   * Normalized chapter/part equivalent. For states that embed agency identity
   * in the citation (FAC), this is the full agency-prefix chapter string.
   * For states where chapter is pure number (TAC, CCR, NYCRR), it is that number
   * cast to string. Always populated; never null for a valid citation.
   *
   * Used as the primary key for structural_rules.chapter_key lookups.
   */
  chapter_key: string;

  /**
   * Normalized section/rule. String to preserve leading zeros (FAC rule "003"
   * must not become integer 3). Null if citation is chapter-level only.
   */
  section_key: string | null;

  // ── Display fields (required for UI rendering) ──────────────────────────────

  /** Human-readable citation exactly as it appears in source material. */
  raw: string;

  /**
   * Cedar's canonical display form. Normalized to a single format per
   * jurisdiction. Used in alert summaries and source attribution.
   *
   * Examples:
   *   FL:  "Fla. Admin. Code R. 64B8-9.003"
   *   TX:  "22 TAC § 174.1"
   *   CA:  "16 CCR § 1399.540"
   *   NY:  "10 NYCRR § 405.2"
   */
  formatted_citation: string;

  /** URL to authoritative source for this citation, if deterministic. Null otherwise. */
  source_url: string | null;

  // ── Classification hints (consumed by stage1Structural) ────────────────────

  /**
   * The agency/board slug inferred from the citation structure itself.
   * FAC: derived from full_division (e.g., '64B8' → 'fl-board-of-medicine')
   * Other states: null — agency must be inferred from metadata or keywords.
   *
   * When non-null, the pipeline can skip agency metadata lookup and assign
   * A-score directly from this slug.
   */
  inferred_agency_slug: string | null;

  /**
   * Confidence that the agency inference is correct. Set by the parser.
   * 1.0 = structurally deterministic (FAC division embedded in citation)
   * 0.7 = high probability (TAC title/part combination narrows to one agency)
   * 0.4 = moderate (CCR title has multiple agencies)
   * 0.0 = no inference possible
   */
  agency_inference_confidence: number;

  /**
   * Whether this citation references an emergency/temporary rule.
   * Affects scoring (emergency rules may warrant higher urgency signals)
   * and display formatting.
   */
  is_emergency_rule: boolean;

  // ── State-specific extension fields ────────────────────────────────────────

  /**
   * Opaque bag of state-specific parsed components. The pipeline never reads
   * these fields directly — they are available to state-specific processors
   * and for debug logging.
   *
   * Examples:
   *   FAC:   { division_letter: 'B', division_number: 8, full_division: '64B8' }
   *   TAC:   { part: 22, chapter: 174, rule: 1 }
   *   CCR:   { division: 13, chapter: 4, article: 7, section: 1399.540 }
   *   NYCRR: { chapter: 1, subchapter: A, part: 405, section: 2 }
   */
  _ext: Record<string, unknown>;
}

type CitationCodeSystem = 'FAC' | 'TAC' | 'CCR' | 'NYCRR' | 'CFR' | string;
// New states add their code_system string to this union.
// 'string' fallback allows unknown states during development without breaking typing.
```

**Design rationale:**

The three pipeline purposes (routing, domain mapping, display) each need different fields. Routing only needs `jurisdiction` and `chapter_key`. Domain mapping needs `chapter_key` and `section_key`. Display needs `formatted_citation` and `source_url`. By separating these cleanly, stages 1–3 never need to know which state parser ran — they consume the same interface.

The `_ext` bag is explicitly opaque to the pipeline. This prevents state-specific logic from leaking into generic pipeline stages. A CA parser can store `{ division: 13, article: 7 }` without requiring the pipeline to understand what CCR divisions are.

`inferred_agency_slug` is the key bridge from citation structure to A-score. For FL, this is always populated (FAC division = agency is structurally deterministic). For TX and CA, it is populated with lower confidence. For NY (NYCRR), agency is often indeterminate from citation alone. The `agency_inference_confidence` field tells the pipeline how much weight to place on the inferred slug vs. falling through to metadata-based agency resolution.

### 1B. Parser Registration Pattern

#### Where implementations live

```
lib/
  citation-parsers/
    index.ts                  ← Registry and dispatcher
    base.ts                   ← Abstract base class
    federal/
      cfr-parser.ts           ← Existing federal parser
    states/

      tx-tac-parser.ts
      ca-ccr-parser.ts
      ny-nycrr-parser.ts
      [state]-[code]-parser.ts  ← Naming convention for new states
```

#### Abstract base class

```typescript
/**
 * StateCitationParser — abstract base for all jurisdiction-specific parsers.
 *
 * Registration contract:
 *   1. Implement jurisdiction (ISO 3166-2 code or 'federal')
 *   2. Implement codeSystem (matches NormalizedCitation.code_system)
 *   3. Implement detectionPatterns (at least one regex)
 *   4. Implement parse()
 *   5. Register via CitationParserRegistry.register() at module load
 */
abstract class StateCitationParser {
  /** ISO 3166-2 subdivision code or 'federal'. */
  abstract readonly jurisdiction: string;

  /** Canonical code system string. Must be unique across all registered parsers. */
  abstract readonly codeSystem: CitationCodeSystem;

  /**
   * One or more regex patterns. The registry uses these to detect which parser
   * should handle a given raw citation string. Patterns are tested in priority
   * order; first match wins. Patterns must be specific enough to avoid collisions
   * (e.g., FAC's 'F.A.C.' context marker vs. CFR's 'C.F.R.' context marker).
   */
  abstract readonly detectionPatterns: RegExp[];

  /**
   * Attempt to parse raw into a NormalizedCitation. Returns null if raw does
   * not match this parser's expected format. Never throws — callers rely on
   * null to fall through to the next candidate parser.
   */
  abstract parse(raw: string): NormalizedCitation | null;

  /**
   * Optional: validate a NormalizedCitation produced by this parser.
   * Called after parse() by the registry to catch structural anomalies
   * (e.g., division number out of known range). Returns null if valid,
   * or a ValidationWarning describing the issue.
   */
  validate?(citation: NormalizedCitation): ValidationWarning | null;

  /**
   * Generate the formatted_citation display string.
   * Subclasses override to apply jurisdiction-specific formatting rules.
   */
  abstract format(citation: NormalizedCitation): string;

  /**
   * Generate the canonical source URL for this citation, if deterministic.
   * Returns null if the URL cannot be constructed from the citation alone.
   */
  abstract sourceUrl(citation: NormalizedCitation): string | null;
}

interface ValidationWarning {
  field: string;
  message: string;
  severity: 'warn' | 'error';
}
```

#### Registry and dispatcher

```typescript
/**
 * CitationParserRegistry — singleton registry for all StateCitationParser
 * implementations.
 *
 * The pipeline calls CitationParserRegistry.getInstance().dispatch(raw, jurisdiction)
 * to get a NormalizedCitation without knowing which parser handled it.
 */
class CitationParserRegistry {
  private static instance: CitationParserRegistry;
  private parsers: Map<string, StateCitationParser> = new Map();
  // key = `${jurisdiction}:${codeSystem}`

  static getInstance(): CitationParserRegistry {
    if (!CitationParserRegistry.instance) {
      CitationParserRegistry.instance = new CitationParserRegistry();
    }
    return CitationParserRegistry.instance;
  }

  /**
   * Register a parser. Called once per parser at module load time.
   * Duplicate registrations (same jurisdiction + codeSystem) throw.
   */
  register(parser: StateCitationParser): void {
    const key = `${parser.jurisdiction}:${parser.codeSystem}`;
    if (this.parsers.has(key)) {
      throw new Error(`Parser already registered for ${key}`);
    }
    this.parsers.set(key, parser);
  }

  /**
   * Main dispatch method. Given a raw citation string and an optional
   * jurisdiction hint, find the correct parser and return a NormalizedCitation.
   *
   * Resolution order:
   *   1. If jurisdiction is known, try only parsers for that jurisdiction,
   *      testing each parser's detectionPatterns against raw.
   *   2. If jurisdiction is null/unknown, test all registered parsers.
   *   3. If no parser matches, return null (caller handles unrecognized format).
   */
  dispatch(raw: string, jurisdiction?: string): NormalizedCitation | null {
    const candidates = jurisdiction
      ? [...this.parsers.values()].filter(p => p.jurisdiction === jurisdiction)
      : [...this.parsers.values()];

    for (const parser of candidates) {
      const matches = parser.detectionPatterns.some(pattern => pattern.test(raw));
      if (matches) {
        const result = parser.parse(raw);
        if (result) {
          const warning = parser.validate?.(result);
          if (warning?.severity === 'error') {
            // Log but do not surface — return null and let pipeline fall through
            console.error(`CitationParser validation error [${parser.jurisdiction}]: ${warning.message}`);
            return null;
          }
          return result;
        }
      }
    }

    return null;
  }

  /**
   * Returns all registered parser metadata. Used by admin tooling to
   * display which jurisdictions are active.
   */
  listParsers(): Array<{ jurisdiction: string; codeSystem: string }> {
    return [...this.parsers.values()].map(p => ({
      jurisdiction: p.jurisdiction,
      codeSystem: p.codeSystem,
    }));
  }
}

// Module-level registration pattern (each parser file registers itself):
// --- fl-fac-parser.ts (bottom of file) ---
// CitationParserRegistry.getInstance().register(new FloridaFACParser());
```

#### Detection pattern specifications for the four initial states

| State | Code System | Primary detection pattern | Notes |
|---|---|---|---|
| FL | FAC | `\d{1,2}[A-Z]{0,2}\d{0,2}-\d{1,3}\.\d{3,5}` (structure) + `F\.A\.C\.` (context) | FAC structure is distinctive — the letter+number division prefix does not appear in other states' codes |
| TX | TAC | `\d{1,2}\s+TAC\s+§\s*\d+\.\d+` | "TAC" abbreviation is unambiguous |
| CA | CCR | `\d{1,2}\s+CCR\s+§\s*\d+(?:\.\d+)?` | "CCR" can collide with unrelated abbreviations; require digit prefix |
| NY | NYCRR | `\d{1,2}\s+NYCRR\s+§?\s*\d+(?:\.\d+)?` | "NYCRR" is unambiguous |

**Pattern collision handling:** If a raw string matches detection patterns for more than one parser (theoretically possible when jurisdiction is unknown), the registry prefers the parser whose `jurisdiction` matches the ingesting source's registered jurisdiction in `source_urls`. The source definition is always the authoritative jurisdiction context.

### 1C. Structural Mapping Configuration Format

The structural mapping is the chapter-key → Cedar domain table. For FL, this is the 150+ chapter mappings from P3-S1 Section 1. Every state needs an equivalent table.

#### Configuration format

```typescript
/**
 * StateStructuralMapping — the complete chapter-level domain mapping for one state.
 * Stored as JSON in the classification_rules table (rule_type = 'structural',
 * jurisdiction = state code) and loaded by loadRuleSet().
 *
 * One StateStructuralMapping per state. Individual entries (ChapterMapping) are
 * the rows that become StructuralRule records in the pipeline.
 */
interface StateStructuralMapping {
  jurisdiction: string;
  code_system: CitationCodeSystem;
  version: string;           // Semver. Increment when mappings change.
  researched_at: string;     // ISO date of last manual research pass
  researcher_notes: string;  // Free text, referenced during onboarding

  /** Ordered list of mappings. Order matters: more specific entries first. */
  chapters: ChapterMapping[];

  /**
   * Chapters that require rule-level decomposition rather than chapter-level
   * assignment. The pipeline flags entities from these chapters for rule-level
   * lookup before assigning a domain.
   */
  rule_level_chapters: string[];  // chapter_keys requiring decomposition

  /**
   * Individual rule-level mappings for chapters listed in rule_level_chapters.
   * Keyed by chapter_key + '.' + section_key.
   */
  rule_overrides: Record<string, ChapterMapping>;
}

interface ChapterMapping {
  /** The chapter_key value this mapping applies to. Matches NormalizedCitation.chapter_key. */
  chapter_key: string;

  /** Human-readable description of what this chapter covers. For documentation only. */
  description: string;

  /** Primary Cedar domain code. */
  primary_domain: string;

  /** Additional domain codes for cross-classification. */
  cross_domains: string[];

  /**
   * Classification reliability at the chapter level.
   * 'high'   = chapter maps cleanly to one domain; no rule-level scan needed
   * 'medium' = chapter is mostly one domain but some rules deviate; keyword scan advised
   * 'low'    = chapter spans multiple domains; rule-level decomposition required
   */
  reliability: 'high' | 'medium' | 'low';

  /**
   * Confidence applied to the structural rule when this mapping matches.
   * High reliability chapters: 1.0
   * Medium: 0.85
   * Low: 0.70 (structural confidence reduced; keywords will adjust)
   */
  structural_confidence: number;

  /** Optional: flags that inform cross-classification trigger evaluation. */
  flags?: ChapterFlag[];
}

type ChapterFlag =
  | 'contains-cs-prescribing'       // May trigger controlled-substances cross-class
  | 'contains-telehealth'            // May trigger telehealth cross-class
  | 'contains-compounding'           // May trigger compounding cross-class
  | 'contains-scope-expansion'       // May trigger state-regulations.scope-expansions
  | 'requires-federal-ref-check'     // Federal reference detection should run
  | 'pain-management-clinic';        // FL/TX pain clinic rules — state-specific facility subtype
```

#### How much is auto-generatable vs. manual

| Component | Auto-generatable | Manual required | Notes |
|---|---|---|---|
| Title-level agency assignments | Partial | Yes | Title numbers and agency names are on the state's official admin code site; a scraper can extract them. But whether a title is "single-purpose healthcare" (A=0.45) vs. mixed (A=0.20) requires human judgment. |
| Chapter list enumeration | Yes (scraper) | No | All state admin code sites publish chapter indexes. A scraper can enumerate `chapter_key` values without human input. |
| Chapter → domain mapping | No | Yes | The chapter description alone is insufficient for accurate domain assignment. Examples: "Standards of Practice" (domain: clinical-operations) vs. "Practice Requirements" (depends on profession). Requires a researcher to read each chapter's subject matter. |
| rule_level_chapters identification | No | Yes | A researcher must identify which chapters span multiple domains. The pattern from FL: ~5% of chapters require rule-level decomposition. |
| rule_overrides | No | Yes | Individual rule decompositions require reading the rules themselves. |
| Cross-domain flags | Partial | Yes | Keywords like "compounding," "telehealth," "pain management clinic" can be detected automatically; the researcher confirms whether the flag is applicable to the chapter's actual content. |
| structural_confidence assignment | Rule-based | Minimal | Follows directly from reliability rating: high→1.0, medium→0.85, low→0.70. Can be computed automatically once reliability is assigned. |

**Expected mapping granularity per state:**

Based on FL (P3-S1) as the reference:

| Metric | FL | Estimated TX | Estimated CA | Estimated NY |
|---|---|---|---|---|
| Total healthcare-relevant chapters | ~150 | ~120 | ~180 | ~200 |
| Chapters requiring rule-level decomposition | ~4 | ~5 | ~8 | ~10 |
| Individual rule overrides needed | ~50 | ~40 | ~80 | ~100 |
| Cross-classification patterns | 18 | ~15 | ~25 | ~20 |
| Hours to research chapter mappings | 16 | 14 | 22 | 24 |

CA and NY are higher because their healthcare code structures are more fragmented (CCR has more healthcare titles; NYCRR Part 400+ spans a larger range of healthcare-adjacent subjects).

## Section 2: State Board Name Normalization

### 2A. Board Name → Cedar Concept Lookup Table Structure

State boards use wildly different naming conventions for equivalent regulatory functions. Cedar needs a lookup table that maps any observed name variant to a stable internal concept.

```typescript
/**
 * BoardNormalizationEntry — one entry in the board name normalization table.
 *
 * Stored in the board_normalization table in Supabase.
 * Loaded by loadRuleSet() alongside classification rules.
 */
interface BoardNormalizationEntry {
  id: string;           // UUID

  /**
   * Cedar's internal board concept identifier. Stable across states.
   * A single concept has one entry per state variant.
   *
   * Convention: {domain-area}.{function}
   * Examples:
   *   'medicine.licensing-board'
   *   'pharmacy.licensing-board'
   *   'nursing.licensing-board'
   *   'mental-health.licensing-board'
   *   'facilities.health-agency'
   *   'medicaid.state-agency'
   *   'controlled-substances.state-program'
   */
  cedar_concept: string;

  /**
   * The jurisdiction this name variant belongs to.
   * 'federal' for federal agencies; state code for state variants.
   */
  jurisdiction: string;

  /**
   * The canonical name Cedar uses in UI display for this jurisdiction.
   * One per (cedar_concept, jurisdiction) pair.
   */
  canonical_name: string;

  /**
   * All known name variants for this board in this jurisdiction.
   * Matched case-insensitively; partial string matching supported.
   * Include historical names (boards get renamed).
   */
  name_variants: string[];

  /**
   * URL patterns that indicate this board's documents.
   * Matched against source_urls.url. Substring match.
   *
   * Example for FL Board of Medicine:
   *   ['floridasmedboard.gov', 'flmedboard.org', 'floridahealth.gov/licensing-and-regulation/medical-doctors']
   */
  url_patterns: string[];

  /**
   * The primary Cedar domain code associated with this board's rulemaking output.
   * Used as classification hint when board is identified by name but citation
   * is unavailable (e.g., plain-text documents, board meeting minutes).
   */
  primary_domain: string;

  /**
   * A-score for this board in this jurisdiction.
   * Duplicates data in classification_rules.agency_rules, but stored here
   * so board name normalization and A-score assignment happen in one lookup.
   */
  a_score: number;

  /**
   * Whether this board has a FL equivalent.
   * null = not yet assessed
   * true = FL has an equivalent board (used to set baseline mapping effort)
   * false = no FL equivalent; taxonomy gap assessment required
   */
  has_fl_equivalent: boolean | null;

  /**
   * If this board has no FL equivalent, the nearest Cedar domain or concept
   * it maps to. Populated during onboarding research.
   */
  nearest_concept: string | null;

  /**
   * When true, this entry was auto-created by the discovery system and has
   * not been manually reviewed. Triggers a HITL task for the researcher.
   */
  needs_review: boolean;

  created_at: string;
  updated_at: string;
}
```

**SQL table definition:**

```sql
CREATE TABLE board_normalization (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  cedar_concept text NOT NULL,
  jurisdiction text NOT NULL,
  canonical_name text NOT NULL,
  name_variants text[] NOT NULL DEFAULT '{}',
  url_patterns text[] NOT NULL DEFAULT '{}',
  primary_domain text NOT NULL,
  a_score numeric(4,2) NOT NULL CHECK (a_score IN (0.00, 0.05, 0.10, 0.15, 0.20, 0.25, 0.30, 0.35, 0.40, 0.45)),
  has_fl_equivalent boolean,
  nearest_concept text,
  needs_review boolean NOT NULL DEFAULT false,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),

  UNIQUE (cedar_concept, jurisdiction)
);

CREATE INDEX idx_board_norm_jurisdiction ON board_normalization (jurisdiction);
CREATE INDEX idx_board_norm_concept ON board_normalization (cedar_concept);
-- GIN index for array membership lookups (name_variants, url_patterns)
CREATE INDEX idx_board_norm_variants ON board_normalization USING GIN (name_variants);
CREATE INDEX idx_board_norm_urls ON board_normalization USING GIN (url_patterns);
```

### 2B. Board Name Detection Heuristics

Three detection pathways, applied in priority order:

**Pathway 1: Source URL pattern matching (highest confidence)**

When a document is ingested from a registered source, the `source_urls` table entry carries a `board_concept` foreign key. The pipeline inherits the board identity directly from the source registration — no text parsing required. This is the expected path for 90% of structured sources (board websites, admin code feeds).

```typescript
// In source ingestion:
const boardEntry = await supabase
  .from('board_normalization')
  .select('cedar_concept, primary_domain, a_score')
  .eq('jurisdiction', source.jurisdiction)
  .contains('url_patterns', [new URL(source.url).hostname])
  .single();
// boardEntry is the deterministic board identity for this source
```

**Pathway 2: Metadata field matching (medium confidence)**

Document metadata often includes an explicit agency/board name (Federal Register has `agencies[]`; FL Admin Register has `rulemaking_authority`; TX Register has `proposed_by`). Match these against `name_variants` case-insensitively.

```typescript
function detectBoardFromMetadata(
  metadata: Record<string, string>,
  jurisdiction: string
): BoardNormalizationEntry | null {
  const fieldPriority = ['rulemaking_authority', 'proposed_by', 'agency', 'author', 'source'];

  for (const field of fieldPriority) {
    const value = metadata[field];
    if (!value) continue;

    const normalized = value.toLowerCase().trim();
    const entries = boardNormalizationCache.filter(e => e.jurisdiction === jurisdiction);

    for (const entry of entries) {
      const variantMatch = entry.name_variants.some(v =>
        normalized.includes(v.toLowerCase()) || v.toLowerCase().includes(normalized)
      );
      if (variantMatch) return entry;
    }
  }

  return null;
}
```

**Pathway 3: Document body text heuristics (lowest confidence, fallback only)**

Used when pathways 1 and 2 fail (unstructured web scrapes, PDF documents). Scan the first 500 characters for board name patterns.

```typescript
const BOARD_MENTION_PATTERNS: RegExp[] = [
  // "The Board of Medicine" / "The Florida Board of Pharmacy"
  /\b(?:the\s+)?(?:[A-Z][a-z]+\s+)?board\s+of\s+[A-Z][a-z][\w\s]+(?:Medicine|Pharmacy|Nursing|Chiropractic|Acupuncture|Dentistry|Optometry)/i,
  // "Texas Medical Board" / "Medical Board of California"
  /\b(?:[A-Z][a-z]+\s+){1,3}(?:medical|pharmacy|nursing)\s+board\b/i,
  // "Office of Professional Medical Conduct" (NY-style)
  /\boffice\s+of\s+professional\s+(?:medical\s+)?conduct\b/i,
  // "Division of Medical Quality Assurance" (FL-style)
  /\bdivision\s+of\s+(?:medical\s+quality\s+assurance|health\s+quality)/i,
];
```

When pathway 3 fires, set `agency_inference_confidence = 0.4` on the resulting citation. This is below the threshold for automated A-score assignment without human review.

### 2C. New Board Discovery — No FL Equivalent

When Cedar encounters a board in a new state that has no FL equivalent, the following protocol applies:

**Step 1: Auto-detection trigger**

A board is flagged as "no FL equivalent discovered" when:
- A source URL or metadata produces a `cedar_concept` match attempt that returns null after searching all registered entries for the target jurisdiction
- The board name passes basic healthcare relevance filters (name contains healthcare-adjacent terms) but does not map to any existing `cedar_concept`

```typescript
// In source registration workflow:
const conceptMatch = await lookupBoardConcept(boardName, jurisdiction);
if (!conceptMatch) {
  await createDiscoveredBoardEntry(boardName, jurisdiction);
  // Triggers HITL task for researcher
}
```

**Step 2: Auto-create provisional entry**

```typescript
async function createDiscoveredBoardEntry(boardName: string, jurisdiction: string): Promise<void> {
  // Attempt nearest-concept heuristic
  const nearestConcept = inferNearestConcept(boardName);

  await supabase.from('board_normalization').insert({
    cedar_concept: `discovered.${slugify(boardName)}`,  // Provisional concept ID
    jurisdiction,
    canonical_name: boardName,
    name_variants: [boardName],
    url_patterns: [],
    primary_domain: nearestConcept?.domain ?? 'state-regulations.licensure',  // Safe fallback
    a_score: 0.25,  // Conservative until reviewed
    has_fl_equivalent: false,
    nearest_concept: nearestConcept?.concept ?? null,
    needs_review: true,  // HITL queue trigger
  });
}

/**
 * Nearest-concept inference heuristics.
 * Uses board name keywords to guess the closest Cedar concept.
 * Confidence is deliberately low — this is a starting point for researcher review.
 */
function inferNearestConcept(boardName: string): { concept: string; domain: string } | null {
  const name = boardName.toLowerCase();

  if (name.includes('naturopath'))
    return { concept: 'naturopathic-medicine.licensing-board', domain: 'state-regulations.licensure' };
  if (name.includes('homeopath'))
    return { concept: 'homeopathy.licensing-board', domain: 'state-regulations.licensure' };
  if (name.includes('midwi'))
    return { concept: 'midwifery.licensing-board', domain: 'state-regulations.licensure' };
  if (name.includes('massage'))
    return { concept: 'massage-therapy.licensing-board', domain: 'state-regulations.licensure' };
  if (name.includes('athletic train'))
    return { concept: 'athletic-training.licensing-board', domain: 'state-regulations.licensure' };
  if (name.includes('dietit') || name.includes('nutrition'))
    return { concept: 'dietetics.licensing-board', domain: 'state-regulations.licensure' };
  if (name.includes('podiat'))
    return { concept: 'podiatry.licensing-board', domain: 'state-regulations.licensure' };
  if (name.includes('veterinar'))
    return null;  // Out of scope; flag and skip
  if (name.includes('funeral') || name.includes('mortuar'))
    return null;  // Out of scope; flag and skip

  return null;
}
```

**Step 3: Researcher HITL task**

A HITL task is created with:
- Board name and jurisdiction
- Nearest concept guess (with confidence = provisional)
- Links to the board's official website
- Checklist: (a) Is this board in scope for Cedar's medical practice target market? (b) Does an existing Cedar concept cover it? (c) If not, should a new concept be added to the taxonomy? (d) What A-score is appropriate?

**Step 4: Taxonomy gap vs. auto-create decision**

| Board type | Decision | Rationale |
|---|---|---|
| Board for a profession Cedar already has a concept for (naturopathic medicine, midwifery, massage therapy) | Add to `cedar_concept` vocabulary; map to `state-regulations.licensure` | These are standard medical profession boards; the domain is the same even if FL has no equivalent |
| Board for a profession outside Cedar's target market (funeral directors, cosmetology, engineering) | Flag as out-of-scope; set `a_score = 0.0`; exclude from rule loading | Cedar serves medical practices; non-medical professional boards should be excluded |
| Board that is actually a sub-unit of an existing board (e.g., "Podiatric Medical Association of [State]" — not a regulatory board) | Flag as non-governmental; exclude | Cedar tracks regulatory bodies, not professional associations |
| Genuinely novel regulatory structure with no obvious analog (rare) | Researcher creates new `cedar_concept`; adds L2 subdomain to taxonomy if needed | This is a taxonomy expansion decision; document in research output |

### 3A. Jurisdiction on the Knowledge Graph Entity

The `jurisdiction` field lives in three places in the Cedar schema, and each has a distinct purpose:

```
source_urls.jurisdiction          → Where the source lives; set at source registration time
changes.jurisdiction              → The jurisdiction of the change event; inherited from source
kg_entities.jurisdiction          → The jurisdiction scope of the knowledge graph entity
```

For `kg_entities`, the current schema has a `jurisdiction` column with default `'FL'` (per CLAUDE.md). This default is intentional for MVP but will need to become explicit in multi-state operation.

**Proposed kg_entities jurisdiction field behavior:**

```sql

-- For multi-state: the default should become an error in production
-- to force explicit jurisdiction assignment at write time.
-- Keep 'FL' default for MVP; set to required after CA onboarding.

-- Federal entities use jurisdiction = 'federal'.
-- An entity that applies to multiple states (e.g., a DEA rule that FL cites)
-- is stored once as jurisdiction = 'federal', not duplicated per state.
-- State-specific rules that implement federal requirements are stored
-- under the state jurisdiction, with a federal_parent_id reference.
```

**Proposed extension for multi-state:**

```sql
ALTER TABLE kg_entities ADD COLUMN IF NOT EXISTS
  federal_parent_id uuid REFERENCES kg_entities(id);
-- When a state entity is a direct implementation of a federal regulation,
-- link it here. Enables cross-jurisdiction impact queries:
-- "Show all state rules implementing 21 CFR Part 1306"
```

### 3B. Jurisdiction Determination at Ingestion Time

Jurisdiction assignment happens in a three-step waterfall:

**Step 1: Source definition (always first)**

Every registered source in `source_urls` has an explicit `jurisdiction` column. This is the primary authority. When a document is ingested from a source, the entity inherits the source's jurisdiction. No parsing or inference is needed.

```typescript
// In monitorSource Inngest function:
const { jurisdiction } = await getSourceDefinition(sourceId);
// jurisdiction is always present; registration rejects sources without it
```

**Step 2: Citation parser output (secondary confirmation)**

If the source jurisdiction is `'FL'` and the citation parser produces a `NormalizedCitation` with `jurisdiction = 'CA'`, this is a conflict. It most often indicates a Florida rule that cross-references California law as context. Resolution:

- Use source jurisdiction as canonical (the entity belongs to the source's regulatory system)
- Store the cross-referenced jurisdiction in `kg_entities.cross_reference_jurisdictions text[]`
- Flag for HITL if the conflict score is high (citation jurisdiction ≠ source jurisdiction with high parser confidence)

**Step 3: Unstructured content fallback**

For web-scraped content without citation references (board meeting minutes, press releases), jurisdiction is inferred from:
1. Source domain → `board_normalization.url_patterns` → `jurisdiction`
2. Keyword presence: "Florida Statutes," "F.S.," "F.A.C." → `'FL'`
3. Address or phone area code in document header (weak signal; use only if steps 1-2 fail)

If none of the three fallbacks succeed, `jurisdiction = source.jurisdiction` is used (the source's registration jurisdiction is always available as the ultimate fallback).

### 3C. Jurisdiction in Search and Filtering

The pipeline and dashboard need three jurisdiction filter modes:

```typescript
type JurisdictionFilter =
  | { mode: 'single'; jurisdiction: string }
  // Example: { mode: 'single', jurisdiction: 'FL' }
  // Returns only FL entities

  | { mode: 'federal_plus_state'; jurisdiction: string }
  // Example: { mode: 'federal_plus_state', jurisdiction: 'FL' }
  // Returns federal entities + FL entities
  // This is the standard practice owner view: all regulations that apply to their FL practice

  | { mode: 'multi_state'; jurisdictions: string[] }
  // Example: { mode: 'multi_state', jurisdictions: ['FL', 'TX', 'CA'] }
  // For practices operating in multiple states

  | { mode: 'all' }
  // Admin/researcher view: all jurisdictions
```

**SQL query pattern:**

```sql
-- Single state mode:
WHERE jurisdiction = 'FL'

-- Federal + state mode (standard practice owner query):
WHERE jurisdiction IN ('federal', 'FL')

-- Multi-state mode:
WHERE jurisdiction IN ('federal', 'FL', 'TX', 'CA')

-- Combined with domain filter:
WHERE jurisdiction IN ('federal', 'FL')
  AND primary_domain LIKE 'controlled-substances%'
```

**Rule loading in the pipeline:**

```typescript
// P2-S1's loadRuleSet extended for multi-state:
async function loadRuleSet(jurisdiction: string): Promise<RuleSet> {
  const { data: rules } = await supabase
    .from('classification_rules')
    .select('*')
    .in('jurisdiction', ['federal', jurisdiction])  // Always load federal + target state
    .order('priority', { ascending: true });

  return buildRuleSet(rules, jurisdiction);
}
```

**Jurisdiction precedence in classification:**

When a federal rule and a state rule both apply to an entity, the state rule's classification takes precedence for the `primary_domain` assignment if:
- The entity is sourced from a state source (jurisdiction = state code)
- The state chapter mapping has reliability = 'high' or 'medium'

Federal rules apply as cross-classifications (with dampened confidence) when the state entity references federal law (using the federal reference detection patterns from P3-S1 Section 5).

## Summary

This framework makes state expansion a bounded, repeatable engineering task. The key architectural decisions:

1. **`NormalizedCitation` as the universal pipeline envelope** — every state parser produces the same interface, so pipeline stages 1–5 never need state-specific logic. State-specific information is in `_ext`, accessible only to state-specific processors.

2. **`CitationParserRegistry` with detection-pattern dispatch** — parsers self-register at module load; the pipeline dispatches by testing patterns against raw citation strings. Adding a state means adding one file to `lib/citation-parsers/states/`; no pipeline code changes.

3. **`StateStructuralMapping` as the configuration boundary** — all chapter-domain knowledge lives in JSON configuration, not code. Updating a domain mapping is a database write, not a deployment.

4. **`BoardNormalizationEntry` as the agency bridge** — arbitrary board names map to stable Cedar concepts through the normalization table, decoupling the source naming from the classification system.

5. **50-hour central estimate, ~72% transfer** — the onboarding effort is concentrated in Phase 2 (structural mapping, ~14–24 hours), which is inherently manual. Everything else is either transferred from the federal + FL base or automatable via seed scripts.

6. **Jurisdiction filter mode `federal_plus_state`** — the standard practice owner view always loads federal rules alongside state rules. This is the semantic contract: Cedar shows a practice what applies to them, not just what their state board published.
