## Section 2: Bill Subject Code → Domain Mapping

### 2.1 FL Legislature Subject Index Structure

The FL Senate publishes a session-end Subject Index PDF listing all bills alphabetically by subject heading. This is a **post-session artifact**, not a real-time per-bill tag. It is useful for historical corpus analysis but not for real-time classification of newly filed bills.

The Subject Index uses **free-text subject headings** (e.g., "Health Care, Providers," "Controlled Substances, Prescriptions," "Pharmacies, Permits"). These are not standardized codes — they are narrative descriptors assigned by legislative staff when bills are enrolled.

### 2.3 Why Subject Codes Are Insufficient Alone

Because FL subject index headings are:
1. Narrative (not codes), and
2. Assigned post-session (not available at ingestion),

Cedar's classification pipeline for legislative bills **cannot rely on subject headings as a Stage 1 structural signal**. Instead, the pipeline should use:

- **Stage 1 signal:** FL Statute citation extracted from bill text or metadata → map via P3-S1 Section 3.1 table (e.g., "458.xxx FS" → `state-regulations.licensure`)
- **Stage 2 signal:** Committee assignment → committee-signal score (C-score): committee HP or HPP → C-score 0.40; committee HCF → C-score 0.40; committee AHS or HCB → C-score 0.35

  *Reconciliation note: These committee-signal scores are used as composite score contributors in the legislative pipeline context. They are within the 0–0.45 range but are distinct from the agency-specific A-score defined in P1-S3. The "C-score" terminology is used here to avoid conflating committee-based signals with agency-based A-scores.*

- **Stage 3 signal:** Bill title + text keyword analysis → K-score per P2-S1 keyword scoring

## Section 3: The Omnibus Bill Problem

### 3.1 Problem Definition

Florida produces multi-topic healthcare bills at high volume. Examples:
- **"Department of Health" bill** (CS/CS/HB 1299, 2025): Covers nurse title changes, ARNP registration updates, DOH administrative housekeeping, and new certification categories — spanning `state-regulations.licensure`, `state-regulations.scope-expansions`, and `clinical-operations.practice-standards`.
- **"Live Healthy" Act** (2023): Covered GAP licensure, autonomous APRN, telehealth, medical education, and workforce development — spanning 5+ Cedar domains.
- **"Healthcare Practitioner Titles"** (CS/HB 1341, 2025): Covers specialist title restrictions for physicians AND title prohibitions for non-physician practitioners — spanning `state-regulations.licensure` and `state-regulations.scope-expansions`.

The core problem: a single legislative entity in Cedar's knowledge graph may be relevant to a practice owner for multiple independent reasons, and those reasons map to different Cedar domains.

### 3.2 Option Evaluation

#### Option A: Entity-Level Cross-Classification
Represent the bill as a single entity with multiple domain assignments (one primary, up to 3 secondary).

**Pros:** Simple implementation. Aligns with the existing `DomainAssignment[]` array in P2-S1's `StageOutput` interface. One entity in the knowledge graph.

**Cons:** For very broad bills (5+ domains), the "primary domain" designation is arbitrary and potentially misleading to practice owners. A hormone clinic owner cares about the prescribing section; the fact that the bill also covers dental therapists is irrelevant noise. Cross-classification reduces alert precision.

**Implementation complexity:** Low. Already supported by `DomainAssignment[]`.

**UX:** Poor for omnibus bills. Practice owners receive an alert about a bill whose title doesn't match their interest, even though one section is highly relevant.

**Accuracy:** High for 2-domain bills. Degrades significantly for 4+ domain bills.

#### Option B: Section-Level Entities
Parse the bill into sections, create one entity per substantive section, classify each section independently.

**Pros:** Maximum granularity. Each alert is precisely targeted. Eliminates false positives.

**Cons:** Requires structured bill parsing at section level — significantly higher implementation complexity. FL bill text does not have machine-readable section boundaries; sections are formatted as prose paragraphs or numbered amendments. Docling processing needed for PDF extraction. Creates explosion in entity count (a 20-section omnibus bill = 20 entities). Section-level entities have no independent legal identity — they cannot be cited, tracked, or linked without the parent bill context.

**Implementation complexity:** High. Requires section boundary detection, parent-child entity relationship, and separate classification per section.

**UX:** Confusing. Sections have no user-recognizable names. "Section 3 of HB 1299" means nothing to a practice owner.

**Accuracy:** Highest possible precision, but at significant UX and implementation cost.

#### Option C: Hybrid (Bill entity cross-classified + significant sections as sub-entities)
The bill entity carries cross-classification for all domains. For sections that contain substantively independent regulatory content in a Cedar-relevant domain that is NOT the bill's primary domain, create a named sub-entity ("Section re: ARNP prescribing authority") linked to the parent bill.

**Pros:** Preserves bill-level identity for citation and tracking. Surfaces domain-specific sub-entities for targeted alerting. Sub-entities inherit the parent bill's citation, sponsor, status, and effective date via relationship link. Scales proportionally — only complex bills generate sub-entities.

**Cons:** Requires threshold logic to determine which sections merit sub-entity creation. Adds parent-child relationship to the knowledge graph schema. More complex than Option A.

**Implementation complexity:** Medium. Requires: (1) section detection logic (keyword-triggered, not full parse), (2) sub-entity creation in knowledge graph when section score ≥ threshold, (3) parent-child link field in `kg_entities`.

**UX:** Best. Practice owner receives alert about the specific provision that affects their practice, with a link to the full bill for context.

**Accuracy:** High. Primary bill entity captures broad relevance; sub-entities surface specific relevance.

### 3.3 Recommendation: Option C (Hybrid)

**Justification:**

Option C is recommended because it solves the omnibus bill problem at proportional cost. The vast majority of FL healthcare bills (roughly 70%) cover a single domain or 2 closely related domains — these require no sub-entity creation, and Option C degrades gracefully to Option A for simple bills. For the 30% of bills that are multi-topic omnibus legislation, Option C adds sub-entity precision without requiring full section-level parsing infrastructure.

**Implementation boundary:** Sub-entities are only created when a section matches a domain **different from the bill's primary domain** AND the section's keyword score for that secondary domain exceeds 0.40. This prevents sub-entity explosion while catching substantively important cross-domain content.

**Alignment with P2-S1 pipeline:** The `classifications: DomainAssignment[]` array on the bill entity handles multi-domain tagging. The sub-entity creation is a post-classification step in the knowledge graph write layer, not part of the scoring pipeline itself. No modifications to P2-S1's `StageOutput` interface are needed.

**Sub-entity naming convention:** `"{Bill Number} — {Topic}" ` (e.g., "HB 1299 — ARNP Autonomous Practice Standards"). The topic is derived from the section's highest-scoring keyword cluster, not free-form text.

## Section 4: Amendment Tracking

### 4.1 Amendment Entity Classification Rules

#### Domain Inheritance Logic

| Scenario | Classification Rule |
|---|---|
| Amendment modifies a single section of the parent bill | **Inherit** parent bill's primary domain at confidence = parent confidence × 0.80. No additional classification required unless amendment text introduces new keywords. |
| Amendment text introduces keywords from a domain NOT in parent bill's `DomainAssignment[]` | **Classify independently.** Run amendment text through full pipeline. If new domain score ≥ 0.40, create new `DomainAssignment` on the parent bill entity (add to secondary domains) AND on the amendment entity. |
| Amendment replaces a section covering Domain A with content covering Domain B | **New primary domain on amendment entity.** Amendment entity classifies to Domain B. Parent bill entity adds Domain B to secondary classifications. HITL flag: "Amendment changes primary domain — review recommended." |
| Floor amendment that substantively rewrites the bill | Treat as equivalent to a new bill version. Re-run full classification on enrolled text. Update parent entity's classifications. Retain amendment entity as a historical sub-entity. |
| Technical/conforming amendment (title change, cross-reference fix, effective date adjustment) | No reclassification. Mark amendment entity as `classification_type: technical`. Inherit parent domain at confidence = 1.0 (structural). |

#### When Does an Amendment Change the Bill's Classification?

An amendment changes the bill's classification when:
1. Amendment text introduces FL Statute section references not present in the original bill, and those statutes map to a different domain (Stage 1 structural trigger).
2. Amendment text score in a new domain exceeds 0.40 via keyword analysis.
3. Amendment text explicitly removes a provision that was the basis of the bill's primary domain classification (e.g., striking the prescribing section from a prescribing bill).

Cedar detects Scenario 3 via deletion-aware diff processing: if a deleted block contained the primary domain's trigger keywords, flag for HITL review to reassess classification.

#### Amendment Entity → Parent Bill Entity Linking

```
amendment_entity.kg_parent_id = parent_bill_entity.id
amendment_entity.entity_type = "legislative_amendment"
amendment_entity.parent_entity_type = "legislative_bill"
amendment_entity.relation = "amends"
```

The amendment entity is a first-class knowledge graph entity (own `id`, own domain classifications, own alert triggers) with a mandatory `kg_parent_id` foreign key to the parent bill.

**Amendment numbering:** FL Committee Substitutes (CS/) are treated as new bill versions, not amendments. CS/HB 1299 is the committee substitute for HB 1299 — Cedar should update the parent entity's bill text pointer and re-run classification, not create a separate amendment entity.

### 5.2 Library vs. Legislative Watch Placement

**Recommendation: Separate "Legislative Watch" view, distinct from the Regulation Library.**

**Justification:**

The Regulation Library contains enacted, current law — it is the authoritative reference for what practices must comply with today. Bills are potential future law. Mixing them into the Library would:
1. Confuse practice owners who cannot always distinguish "currently in effect" from "pending."
2. Create false compliance pressure from bills that never pass.
3. Dilute the Library's reliability signal.

A separate Legislative Watch view maintains the Library's authority while providing the pre-regulatory intelligence that differentiates Cedar from basic alerting services.

**Legislative Watch characteristics:**
- Bills displayed with prominent status badge (Filed / Committee / Floor / Enrolled / Signed)
- Estimated effective date shown when available
- Link to full text on flsenate.gov or flhouse.gov
- AI summary: "What this bill would change" (present conditional, not present tense)
- Disclaimer: "This bill has not become law. Cedar monitors its progress and will alert you if signed."
- Filter by domain allows targeted legislative tracking

**Intelligence tier only:** Legislative Watch is an Intelligence tier feature. Monitor tier users receive post-signing alerts when bills become effective statutes.

### 5.3 Entity Transition: Bill → Statute

When a bill is signed by the Governor and assigned a Chapter Law number:

1. **Create new Statute entity** in the Regulation Library with:
   - `entity_type: "florida_statute_amendment"`
   - `citation: "Ch. [X], Laws of Florida [Year]" → codified as §[FS section]`
   - `domain_classifications`: inherited from enrolled bill entity (re-run classification on enrolled text for accuracy)
   - `effective_date`: as specified in the bill
   - `source_url`: FL Statutes online or session laws URL

2. **Link** the Statute entity to the originating Bill entity via `superseded_by_legislative` relationship.

3. **Update** the Legislative Watch Bill entity status to `enacted`. Do not delete — preserve legislative history.

4. **Alert sequence:**
   - Alert 1 (when signed): "Governor signed — [Bill Number]. Takes effect [date]."
   - Alert 2 (on effective date): "Now in effect: [Statute section change]." Triggered by scheduled job based on `effective_date`.

5. **FAC rulemaking downstream:** If the statute directs rulemaking ("The Board shall adopt rules..."), Cedar should flag the originating statute entity with `triggers_rulemaking: true`. When the FAC rulemaking notice appears in the Florida Administrative Register, link the new rule entity back to the originating statute entity.

## Section 6: Board Meeting Minutes — Classification Strategy

### 6.1 Entity Granularity Decision

**Evaluation: Document-Level vs. Decision-Level**

| Dimension | Document-Level | Decision-Level |
|---|---|---|
| Entity count | 1 entity per meeting | 5–30 entities per meeting (one per substantive agenda item) |
| Classification accuracy | Lower — a single meeting contains enforcement, licensure, and rulemaking items (3+ domains) | Higher — each decision classifies to a specific domain |
| Alert precision | Low — "Board of Medicine met" is not actionable | High — "Board of Medicine approved new prescribing standard" is actionable |
| Extraction complexity | Low — minimal parsing needed | High — requires agenda item extraction and boundary detection |
| Knowledge graph value | Low — document node has minimal relationship specificity | High — decision nodes connect to practitioners, statutes, and affected domains |
| HITL burden | Low | Medium — more entities require more review |
| Implementation phasing | Quick to implement | Requires more robust parsing infrastructure |

**Recommendation: Decision-Level entities, implemented in two phases.**

**Phase 1 (MVP):** Document-level entities for initial corpus build. One entity per meeting, classified to the board's primary domain. Provides basic awareness signal.

**Phase 2 (post-MVP):** Decision-level extraction. Identify the following agenda item types from the minutes text, create sub-entities per item:
- Enforcement/disciplinary actions
- License grants, denials, and modifications
- Rulemaking proposals and adoptions
- Policy interpretations
- Waiver/exception approvals

**Justification:** Decision-level classification is architecturally superior and required for the knowledge graph to be useful. The phased approach allows initial corpus population without requiring full parsing infrastructure at launch. The Phase 2 transition is incremental — the document-level entity becomes the parent, and decision-level sub-entities are added as parsing improves.

### 6.4 Confidence Calibration for Board Minutes

**Why minutes accuracy is structurally lower than regulatory text:**

1. Minutes are retrospective narrative summaries, not structured regulatory text. They lack citations, statute references, and the formal FAC citation structure that enables structural classification.
2. Minutes mix procedural content (quorum calls, motion votes, minute approvals) with substantive regulatory decisions. Procedural content generates false classification signals.
3. A single enforcement action may be described in one sentence or three paragraphs depending on the board's meeting style.
4. Decision type vocabulary is not standardized across boards (see Section 7 on per-board variation).

**Realistic accuracy targets:**

| Classification Level | Federal Regulatory Text Target | FL Board Minutes Target | Gap Analysis |
|---|---|---|---|
| L1 domain (broad category) | 92–95% | 80–88% | Minutes lack FAC citations; committee names serve as partial proxy |
| L2 subdomain | 80–85% | 60–70% | Decision type mapping (Section 6.2) provides primary signal; text ambiguity reduces confidence |
| Decision type | N/A | 70–80% | Decision type labels are a new classification layer with limited training signal |

**These are expected accuracy ranges under deterministic + keyword classification without AI refinement.** With AI stage (Stage 4 in P2-S1 pipeline), L2 accuracy for minutes can reach 72–80%.

**Recommended confidence threshold for HITL flagging:**

| Score Range | Action |
|---|---|
| ≥ 0.60 | Classify and deliver. No HITL required unless in "Always HITL" category per Section 6.2. |
| 0.40–0.59 | Flag to HITL queue. Deliver with confidence caveat: "Classification confidence is moderate — AI summary pending reviewer confirmation." |
| < 0.40 | Hold for HITL. Do not deliver until reviewed. |

**Note:** The HITL confidence threshold for board minutes is the same as the general pipeline threshold. What changes is the expected distribution: for regulatory text, ~70% of entities score ≥ 0.60. For board minutes, expect ~45–55% to score ≥ 0.60, placing roughly half in the HITL queue. This is expected and acceptable given the higher ambiguity of minutes content.

**Categories that always require HITL review (regardless of score):**
1. Scope of practice decisions or interpretations
2. Policy interpretations and declaratory statements (guidance documents)
3. Waiver or exception approvals that may signal enforcement direction shifts
4. Joint/cross-board actions
5. Any action described with legal standard language ("precedent," "declaratory statement," "advisory opinion")
6. Enforcement actions in Cedar's target practice types (functional medicine, hormone optimization, compounding, med spa) — high practice-owner impact warrants human review

### 7.2 Structural Patterns: Board of Medicine

Verified against meeting archives at flboardofmedicine.gov.

**Document set per meeting:** Notice (PDF), Agenda (PDF), Minutes (PDF), Audio recording.

**Minutes structure — standard section sequence:**
1. Call to Order / Quorum Determination
2. Approval of Previous Meeting Minutes
3. General Business
   - Correspondence items
   - Legal/DOH staff reports
4. Disciplinary Cases (quasi-judicial proceedings; largest section by page count)
   - Case docket items: license number, respondent name, charge summary, action taken
   - Format: tabular docket list + prose summary for contested cases
5. Licensure Items
   - Applications approved/denied/deferred (often as a consent agenda list)
   - Special licensure categories (temporary certificates, GAP licenses)
6. Rulemaking
   - Notices of proposed rule development
   - Adoption votes
   - Workshop scheduling
7. Committee Reports (standing committees: Probable Cause Panel, Standards of Practice committee)
8. Policy/Correspondence Items (petitions, declaratory statements, board-to-board letters)
9. Adjournment

**Cedar-relevant extraction targets:** Sections 4, 5, 6, 8. Sections 1, 2, 3, 7, 9 are procedural and generate low-value content for classification.

### 7.3 Structural Patterns: Board of Pharmacy

Verified against floridaspharmacy.gov archive (Feb. 2021 minutes PDF and site structure).

**Document set per meeting:** Minutes (PDF), Audio, Agenda.

**Meeting types (two distinct meeting types per cycle):**
- **Probable Cause Panel Meeting** — Scheduled before or on the same day as the General Business Meeting. Reviews complaints to determine if sufficient cause exists to file an Administrative Complaint. This is a separate docket from disciplinary hearings.
- **General Business Meeting** — Full board meeting with all agenda categories below.

**Minutes structure — General Business Meeting:**
1. Call to Order
2. Approval of Previous Minutes
3. General Business / Correspondence
4. **Application Reviews** — Pharmacy permit applications, pharmacist licensure, approval/denial lists. Often cross-board (pharmacy + medicine + osteopathic joint review for certain approvals). Includes a "Joint Rules Committee" vote notation.
5. **Rulemaking** — Rule development, workshops, adoption votes. Active chapter: 64B16.
6. **Disciplinary/Enforcement Docket** — Findings of fact, final orders, probation conditions, compliance appearances. Includes mandatory CE/training requirements as part of discipline.
7. Committee Reports
8. Adjournment

**Key structural difference from Board of Medicine:** The Board of Pharmacy explicitly labels a "Probable Cause Panel" as a separate meeting entity, not a section within the General Business minutes. Cedar should track Probable Cause Panel meeting minutes as a distinct document type with its own classification context — these are investigative proceedings, not final enforcement actions. Classification: `state-regulations.licensure` (process domain) with domain cross-classification matching the complaint subject.

### 7.4 Structural Patterns: AHCA

AHCA does not operate a "Board" in the same sense as the professional licensing boards under DOH/MQA. AHCA's regulatory meetings fall into two categories:

**1. Advisory Committee Meetings (Pharmaceutical & Therapeutics Committee, Drug Utilization Review Board):**
- These committees advise AHCA on Medicaid drug coverage policy.
- Minutes cover: drug coverage recommendations, prior authorization criteria changes, preferred drug list updates, drug utilization review findings.
- **Cedar domain:** `medicare-billing.*` (coverage determinations, preferred drug lists) with cross-classification to `compounding.*` or `controlled-substances.*` when specific drug categories are discussed.
- Format: Structured agendas, committee-style minutes, posted on ahca.myflorida.com/medicaid.

**2. Rulemaking Workshops (59A, 59G, 59C chapters):**
- AHCA holds public workshops for proposed rules.
- Minutes and transcripts document testimony and AHCA responses.
- **Cedar domain:** Matches the FAC chapter under rulemaking per P3-S1 Section 1.10 tables.
- Format: Workshop transcripts (not traditional minutes), posted on flrules.org as part of the rulemaking docket.

**AHCA has no equivalent to the DOH board "Probable Cause Panel."** Enforcement actions against AHCA-licensed facilities (59A-33 health care clinic enforcement, Medicaid fraud) are handled through the Division of Medicaid Fraud Control (FDLE) or AHCA's survey and certification process, which generates a different document type (survey reports, enforcement letters) — not meeting minutes.

### 7.5 Formatting Variations: Consistency Assessment

| Element | Board of Medicine | Board of Pharmacy | AHCA Advisory Committees |
|---|---|---|---|
| Document format | PDF, consistently structured | PDF, consistently structured | PDF, varies by committee |
| Section headers | Consistent across meetings | Consistent within meeting type | Varies by committee |
| Docket item format | License number + case caption | License number + violation citation + disposition | Drug name + recommendation + vote |
| Rulemaking notation | FAC chapter + rule number cited | FAC chapter + rule number cited | Rule chapter cited, less formal |
| Cross-board notation | Explicit when joint actions occur | Explicit ("Joint Rules Committee") | N/A |
| Procedural vs. substantive ratio | ~30% procedural / 70% substantive | ~25% procedural / 75% substantive | ~20% procedural / 80% substantive |
| Machine-readability of PDFs | Text-layer PDFs (searchable) | Text-layer PDFs (searchable) | Varies — some are image-scanned |

**Consistency finding:** The Board of Medicine and Board of Pharmacy follow highly consistent internal formats — section headers, docket item structure, and terminology are stable across meetings within each board. This enables pattern-based extraction rules tailored per board. AHCA advisory committees show more variation and require per-committee extraction rules.

**Inconsistency across boards:** The two boards use different terminology for equivalent concepts. Board of Medicine uses "Disciplinary Cases." Board of Pharmacy uses "Enforcement Docket" and "Probable Cause Panel" as distinct entities. A Cedar classification rule that looks for "Disciplinary Cases" as a section header would miss Board of Pharmacy enforcement content. Classification rules must be **board-specific**, not shared.

**Practical extraction approach:** Extract using a **board-specific section header map** that identifies the sections in Section 7.2 and 7.3 above, then classifies content within each section using the decision type → domain mapping in Section 6.2. The board identity (known from the source document's URL/metadata) sets the default domain context before any text analysis begins — this is the minutes equivalent of P3-S1's A-score advantage.
