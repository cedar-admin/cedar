# Real-Time Entity Classification, Taxonomy Evolution & Alerting

## Overview

This document defines Cedar's operational protocols for three interconnected concerns:

1. **Real-time single-entity classification** — how the batch-oriented P2-S1 pipeline adapts when a new regulatory entity arrives from a live source during ongoing monitoring
2. **Taxonomy gap detection and evolution** — how gaps in the existing 11-L1/55-L2/~320-L3–L6 taxonomy are detected, surfaced, and resolved without breaking backward compatibility
3. **Alerting rules** — implementable Inngest-function-level rules for confidence, anomaly, volume, and conflict alerts
4. **Extended monitoring metrics** — five new metrics supplementing P2-S5's ten, covering non-federal sources, real-time mode, and taxonomy tracking

All rules in this document are written to be implementable as Inngest functions, database triggers, or cron jobs with no further research required.

## Part 1: Real-Time Single-Entity Classification Flow

### 1A. Pipeline Adaptation for Single-Entity Mode

The P2-S1 pipeline was designed for batch classification of the initial ~99K corpus. In that mode: concurrency is 10, batch size for Stage 4 AI is 3, and throughput is optimized over per-entity latency. In real-time mode, one entity arrives from a live monitoring source and must be classified and routed to the user-facing Library within a defined latency budget.

**Latency budget**: Target ≤ 8 minutes from ingestion to Library appearance for ≥ 0.85 confidence entities. Entities entering HITL review have no user-facing latency commitment (they appear only after reviewer action).

**Stage behavior in single-entity mode**:

| Stage | Batch Mode | Single-Entity Mode | Δ Notes |
|---|---|---|---|
| 1: Structural | Always runs | Always runs | No change; purely deterministic, ~50ms |
| 2: Metadata | Always runs if Stage 1 misses | Always runs if Stage 1 misses | No change; ~50ms |
| 3a: Keywords | Always runs | Always runs | No change; ~100ms |
| 3b: Embeddings | Deferred (cost) | Deferred (not activated yet) | Activate per P2-S5 threshold: >150 web entities/day |
| 4: AI Classifier | Batch size 3, Haiku | **Batch size 1**, Haiku | Higher per-entity cost ($0.0048 vs $0.0023); unavoidable |
| 5: Irrelevance | Batch size 20, Haiku | **Batch size 1**, Haiku | Only for low-confidence entities |

**Short-circuit rules** (stop the pipeline as soon as high confidence is achieved):

```
SHORT_CIRCUIT_1: Stage 1 returns composite_score >= 1.0 AND classifications.length > 0
  → Skip Stages 2, 3a, 4, 5
  → Persist classification immediately
  → Target latency: < 200ms

SHORT_CIRCUIT_2: Stage 2 early resolution: A+D+B >= 0.60 AND no excluded_agency signal
  → Skip Stages 3a, 4, 5
  → Persist classification immediately
  → Target latency: < 400ms

SHORT_CIRCUIT_3: Stage 3a composite score >= 0.60
  → Skip Stages 4, 5
  → Persist classification immediately
  → Target latency: < 600ms

NO SHORT_CIRCUIT: Composite score < 0.60 after Stage 3a → Must run Stage 4 (AI)
  → Target latency: < 8 minutes (includes AI API call + retry headroom)
```

**Stage 4 single-entity cost**:
- Input tokens: system prompt (1,800) + taxonomy context (2,500) + single entity user prompt (~150) = 4,450
- Output tokens: ~300 (single entity response is smaller than batch)
- Cost at Haiku pricing: (4,450 × $0.25 + 300 × $1.25) / 1,000,000 = **$0.00149 per entity**
- Monthly cost for 100 entities/day reaching Stage 4: ~$4.47/month. Acceptable for ongoing operations.

**Inngest function signature for single-entity real-time mode**:

```typescript

```

### 1B. Relevance Gate — Decision Tree

The relevance gate runs before Stage 1. Its purpose is to avoid wasting pipeline compute on entities that are structurally irrelevant before any scoring occurs. The gate uses two inputs from P1-S3: the source allowlist and the source denylist.

```
RELEVANCE GATE DECISION TREE
═══════════════════════════════════════════════════════════════
INPUT: entity arrives with source_slug, jurisdiction

NODE 1: Is source_slug on the DENYLIST?
  → YES: Mark entity irrelevant_at_gate=true, skip pipeline entirely.
         Log to kg_classification_log with s3_tier='irrelevant', stage='gate'.
         Done.
  → NO: Continue to NODE 2.

NODE 2: Is source_slug on the ALLOWLIST with category='high_relevance'?
  (Examples: fl-board-of-medicine, fl-board-pharmacy, dea-registrant-actions)
  → YES: Skip relevance scoring. Set gate_score=1.0 (pre-approved source).
         Enter Stage 1 directly. Skip Stage 5 (irrelevance confirmation).
         → Continue to STAGE 1.

NODE 3: Is source_slug on the ALLOWLIST with category='mixed_relevance'?
  (Examples: federal-register, fl-administrative-register, cms-transmittals)
  → YES: Apply S3 agency/source scoring model first (Stage 2 only).
         Compute A score from source-type mapping.
         If A >= 0.25: gate passes. → Continue to STAGE 1.
         If A < 0.25:  apply keyword pre-screen (50-word title/description check).
                       If any high-weight keyword matches: gate passes. → STAGE 1.
                       If no keyword match: mark needs_relevance_review=true,
                       route to Stage 5 (irrelevance confirmation) before Stage 4.

NODE 4: Source not on either list (new/unknown source)
  → Apply conservative keyword pre-screen on title + description.
     If 2+ strong keyword matches: tentative pass, tag with unknown_source_flag=true.
     If < 2 matches: hold entity, emit 'source/unknown.detected' event for admin review.
     Do not classify until source is categorized.
═══════════════════════════════════════════════════════════════
```

### 1C. Confidence-Based Routing to User-Facing Views

After the pipeline completes, the final confidence score on the primary domain assignment determines how and when the entity appears in the Library.

**Routing logic** (implementable rules, executed in `persistClassification()`):

```
ROUTING RULE R1: High confidence — immediate Library appearance
  Condition: primary_domain.confidence >= 0.85
  Actions:
    - Set classification_status = 'accepted'
    - Set library_visible = true
    - Set review_required = false
    - Emit 'classification/accepted' event → triggers change detection pipeline
    - No visual indicator in Library (entity appears as fully classified)

ROUTING RULE R2: Medium confidence — Library appearance with pending indicator
  Condition: 0.60 <= primary_domain.confidence < 0.85
  Actions:
    - Set classification_status = 'accepted_with_flag'
    - Set library_visible = true
    - Set review_required = true (queued for low-priority HITL)
    - Set review_priority = 'low' (does not block delivery)
    - Emit 'classification/accepted_flagged' event
    - UI indicator: yellow "Classification under review" badge on entity card
    - Indicator is REMOVED when: reviewer accepts, or 30-day auto-acceptance triggers
      (auto-acceptance condition: no reviewer correction after 30 days AND no anomaly alert)

ROUTING RULE R3: Low confidence — Library withheld pending HITL
  Condition: primary_domain.confidence < 0.60
  Actions:
    - Set classification_status = 'pending_review'
    - Set library_visible = false
    - Set review_required = true
    - Set review_priority based on P2-S5's priority formula:
        priority = 0.4×(1-confidence) + 0.3×impact + 0.2×confusion_risk + 0.1×recency
    - Emit 'classification/needs_review' event → routes to HITL queue
    - Entity does NOT appear in user Library until reviewer action
    - No source-authority exception below 0.65 — high-relevance sources at
      confidence < 0.65 go to HITL without Library appearance.

ROUTING RULE R4: Source authority override
  Condition: source gate category = 'high_relevance' AND 0.65 <= confidence < 0.85
  Actions:
    - Override ROUTING RULE R2 (upgrades yellow badge to orange badge)
    - Set library_visible = true
    - Set classification_status = 'accepted_source_override'
    - Display orange "Classification pending review" badge
      (replaces R2's yellow "Classification under review" badge)
    - Set review_priority = 'medium' (higher than R2's 'low')
    - Rationale: a FL Board of Medicine rule at 0.70 confidence warrants a more
      prominent review signal than a generic federal source at the same confidence.
      Source authority elevates urgency, not visibility threshold.
```

**Visual indicator specifications** for the Library UI:

| Status | Badge Text | Color | Removal Condition |
|---|---|---|---|
| `accepted` | None (no badge) | — | N/A |
| `accepted_with_flag` | "Classification under review" | Yellow | Reviewer confirms OR 30-day auto-accept |
| `accepted_source_override` | "Classification pending review" | Orange | Reviewer confirms |
| `pending_review` | Not visible in Library | — | Reviewer approves + sets library_visible=true |

## Part 2: Taxonomy Gap Detection

### 2A. Detection Signals

Four signals indicate a gap in the existing taxonomy. For each signal, the detection logic and expected false positive rate are defined.

---

**Gap Signal G1: L3+ Subdomain Miss**

An entity's AI classification confidently identifies an L1 and L2 domain but cannot assign any L3+ code with confidence >= 0.60.

Detection logic:
```sql
-- Entities where AI assigned L2 but no L3+ with adequate confidence
SELECT
  e.entity_id,
  ecl.domain_code AS l2_assigned,
  ecl.confidence  AS l2_confidence,
  ecl.reasoning
FROM kg_classification_log ecl
JOIN kg_entities e ON e.id = ecl.entity_id
WHERE ecl.taxonomy_level IN ('L1', 'L2')
  AND ecl.confidence >= 0.75         -- confident at L2
  AND ecl.uncertainty_flag = false
  AND NOT EXISTS (
    SELECT 1 FROM kg_classification_log ecl2
    WHERE ecl2.entity_id = ecl.entity_id
      AND ecl2.taxonomy_level NOT IN ('L1', 'L2')
      AND ecl2.confidence >= 0.60
  )
  AND ecl.created_at > NOW() - INTERVAL '30 days';
```

False positive rate expectation: **Low (~10%)**. When an entity is genuinely at L2 granularity (e.g., a broad rulemaking notice covering an entire regulatory domain), classification at L2 is correct. The false positive check: does the entity's text reference a specific regulatory concept that should have an L3 equivalent? If not, it is a true L2-only entity, not a gap.

---

**Gap Signal G2: AI Suggests Nonexistent Domain**

The AI classifier returns a `domain_code` string that does not match any node in the live taxonomy. This can happen when the model hallucinates a plausible-sounding code, or when it correctly identifies a concept that genuinely has no representation.

Detection logic:
```typescript
// In stage4-ai.ts, after JSON parse and before persistClassification:
const knownDomainCodes = await loadTaxonomyCodes(jurisdiction);  // cached, 1-minute TTL

for (const classification of response.classifications) {
  if (!knownDomainCodes.has(classification.domain_code)) {
    await step.run("flag-unknown-domain", async () => {
      await supabase.from("taxonomy_gap_events").insert({
        entity_id,
        signal_type: "nonexistent_domain_suggested",
        suggested_code: classification.domain_code,
        confidence: classification.confidence,
        reasoning: classification.reasoning,
        detected_at: new Date().toISOString(),
        jurisdiction,
      });
    });
    // Fall back: assign to nearest parent by prefix matching
    classification.domain_code = findNearestParent(classification.domain_code, knownDomainCodes);
    classification.confidence = classification.confidence * 0.70;  // discount for fallback
    classification.taxonomy_gap_flag = true;
  }
}
```

False positive rate expectation: **Medium (~30%)**. The most common false positive is a valid code with a typo or formatting variant (e.g., `controlled-substances.prescribing.pdmp` vs `controlled_substances.prescribing.pdmp`). Normalize codes before comparison. True positives are the model correctly naming a concept that does not yet have a code.

---

**Gap Signal G3: Cluster Analysis — L2 Accumulation with Low L3 Confidence**

When multiple recent entities are all classified to the same L2 domain with low average L3 confidence, it suggests that a new regulatory concept is emerging within that L2 area that the existing L3 nodes do not capture.

Detection logic (daily cron):
```sql
-- L2 domains where recent entities cluster with low L3 confidence
SELECT
  l2_domain,
  COUNT(*)                     AS entity_count,
  AVG(max_l3_confidence)       AS avg_l3_confidence,
  MIN(classified_at)           AS window_start
FROM (
  SELECT
    entity_id,
    SUBSTRING(domain_code FROM '^[^.]+\.[^.]+') AS l2_domain,
    MAX(confidence) FILTER (
      WHERE taxonomy_level NOT IN ('L1', 'L2')
    ) AS max_l3_confidence,
    MAX(created_at) AS classified_at
  FROM kg_classification_log
  WHERE created_at > NOW() - INTERVAL '14 days'
  GROUP BY entity_id, SUBSTRING(domain_code FROM '^[^.]+\.[^.]+')
) sub
WHERE max_l3_confidence < 0.65
   OR max_l3_confidence IS NULL
GROUP BY l2_domain
HAVING COUNT(*) >= 5
ORDER BY entity_count DESC;
```

Alert threshold: 5+ entities in the same L2 domain within 14 days with avg L3 confidence < 0.65.

False positive rate expectation: **Medium-high (~40%)** during periods of genuine regulatory activity in a single area. A legitimate burst of Board of Medicine enforcement actions against med spas will all cluster in `state-regulations.licensure` without requiring a new L3 node. Distinguish: does the cluster share common keywords suggesting a new concept, or do they represent volume in an existing concept?

---

**Gap Signal G4: Orphaned Regulatory Concept**

The AI classifier's `reasoning` field mentions a regulatory concept that appears in multiple entities but has no corresponding taxonomy node. This is detected by NLP analysis of reasoning strings rather than code validation.

Detection logic:
```sql
-- Concepts appearing in reasoning strings across multiple entities
-- that are not represented as domain codes (simplified version)
SELECT
  extracted_concept,
  COUNT(DISTINCT entity_id) AS mention_count,
  ARRAY_AGG(DISTINCT l2_domain ORDER BY l2_domain) AS associated_l2_domains
FROM taxonomy_gap_events
WHERE signal_type = 'orphaned_concept'
  AND detected_at > NOW() - INTERVAL '30 days'
GROUP BY extracted_concept
HAVING COUNT(DISTINCT entity_id) >= 3
ORDER BY mention_count DESC;
```

The `orphaned_concept` extraction is done at classification time: a regex/LLM pass over the AI's `reasoning` field extracts noun phrases that look like regulatory concepts (contain words like "rule", "requirement", "standard", "guideline", "permit", "registration", "threshold", "waiver") and do not match any known taxonomy code.

False positive rate expectation: **High (~60%)**. Many reasoning strings reference established regulatory concepts that are intentionally grouped under broader codes. This signal produces the most false positives and should be reviewed in aggregate (only escalate when a concept appears 3+ times across different entities).

### 2B. Gap Response Protocol

**Immediate response** (at classification time, when any gap signal fires):

```
IMMEDIATE PROTOCOL:
1. Classify entity to nearest parent domain (L2 if L3 unavailable, L1 if L2 unavailable)
2. Set taxonomy_gap_flag = true on the kg_entity_domains row
3. Set gap_signal_type = one of: 'l3_miss' | 'nonexistent_code' | 'cluster' | 'orphaned_concept'
4. Store full AI reasoning text in gap_reasoning column (for later re-classification context)
5. Apply confidence discount:
   - L3 miss: no discount (L2 assignment may be fully correct)
   - Nonexistent code: ×0.70 discount on confidence (fallback assignment is uncertain)
   - Orphaned concept: ×0.85 discount
6. Route entity per confidence-based routing rules in 1C above
7. Emit Inngest event: 'taxonomy/gap.detected' with gap_signal_type payload
```

Metadata stored for later re-classification:
- `gap_signal_type` — which signal fired
- `gap_reasoning` — AI's full reasoning text verbatim
- `suggested_code` — the nonexistent code suggested by AI (for G2 events)
- `classified_at_gap` — timestamp, used to scope re-classification when gap is resolved
- `gap_cluster_id` — links related gap events when they share an L2 domain and time window

**Short-term response** (gap aggregation, runs every 4 hours):

Inngest function `taxonomy/gap.aggregate` fires every 4 hours and runs the following checks:

```typescript
// Gap escalation thresholds:
const CLUSTER_THRESHOLD = 5;      // entities in same L2 within 14 days
const CONCEPT_THRESHOLD = 3;      // entities referencing same orphaned concept within 30 days
const CODE_MISS_THRESHOLD = 3;    // same nonexistent code suggested within 7 days

// Admin alert format when threshold exceeded:
interface GapAlert {
  alert_id: string;
  alert_type: 'l2_cluster' | 'repeated_code_miss' | 'orphaned_concept_cluster';
  severity: 'info' | 'warning' | 'critical';
  affected_l2_domain: string;          // e.g., "state-regulations.scope-of-practice"
  entity_count: number;
  suggested_new_code?: string;         // for repeated_code_miss type
  concept_name?: string;               // for orphaned_concept_cluster type
  example_entity_ids: string[];        // first 3 entity IDs in the cluster
  time_window: string;                 // "2026-03-10 to 2026-03-24"
  recommended_action: string;          // human-readable suggestion
  created_at: string;
}
```

Alert delivery: admin panel notification (Bell icon, taxonomy alerts section) + email to `reviewer` role users. No practice-owner delivery — taxonomy gaps are internal operational concerns.

**Severity mapping**:
- `info`: 5–9 entities in cluster OR 3–4 orphaned concept mentions
- `warning`: 10–19 entities OR same nonexistent code suggested 3+ times in 7 days
- `critical`: 20+ entities in cluster within 14 days (indicates a major new regulatory area)

**Resolution workflow**:

```
TAXONOMY GAP RESOLUTION WORKFLOW
═══════════════════════════════════════════════════════════════════
STEP 1: Admin or reviewer reviews gap alert in admin panel.
        Views: affected entities, AI reasoning texts, suggested codes.

STEP 2: Admin proposes new taxonomy node via admin UI:
        - Proposed domain code (e.g., "state-regulations.scope-of-practice.med-spa-oversight")
        - Description (required, min 50 characters)
        - Parent node (must be existing L1 or L2)
        - Classification signals: keywords, agencies, statutes (min 3 keywords required)
        Proposal written to: taxonomy_proposals table with status='proposed'

STEP 3: (Optional) second reviewer approves proposal.
        If proposal is accepted: status → 'approved'
        If proposal is rejected: status → 'rejected' with rejection_reason

STEP 4: On approval, taxonomy node is created:
        INSERT INTO taxonomy_nodes (...) with new code, parent_code, version
        Emits event: 'taxonomy/node.created' with new_code payload

STEP 5: Re-classification sweep triggered automatically (see Part 3C)
        All entities with taxonomy_gap_flag=true AND gap_l2_domain = new node's parent
        are re-queued for Stage 4 AI classification with updated taxonomy context.

STEP 6: After re-classification, taxonomy_gap_flag cleared on updated entities.
═══════════════════════════════════════════════════════════════════
```

**Who can add new domains**: Only users with `reviewer` or `admin` role. `practice_owner` role has no taxonomy management access.

## Part 3: Taxonomy Evolution

### 3A. Taxonomy Versioning

Cedar uses **sequential version numbers per node, tracked in a `taxonomy_versions` audit log**. This is simpler than timestamped migration files for a tree structure and directly queryable.

The live taxonomy is always the current state of `taxonomy_nodes`. Changes to nodes generate rows in `taxonomy_versions` (immutable log). Entities reference the node code (string), not a version ID, so they do not need to be updated on simple renames within the same code — only on structural changes.

**Schema DDL**:

```sql
-- The live taxonomy tree
CREATE TABLE taxonomy_nodes (
  code                TEXT PRIMARY KEY,   -- e.g., "controlled-substances.prescribing.pdmp"
  parent_code         TEXT REFERENCES taxonomy_nodes(code),
  label               TEXT NOT NULL,
  description         TEXT,
  level               SMALLINT NOT NULL CHECK (level BETWEEN 1 AND 8),
  classification_signals JSONB,          -- keywords, agencies, cfr_ranges, statutes
  jurisdiction        TEXT NOT NULL DEFAULT 'federal',
  status              TEXT NOT NULL DEFAULT 'active'
                        CHECK (status IN ('active', 'deprecated', 'merged_into', 'split_from')),
  merged_into_code    TEXT REFERENCES taxonomy_nodes(code),  -- set when status='merged_into'
  split_from_code     TEXT REFERENCES taxonomy_nodes(code),  -- set when status='split_from'
  node_version        INTEGER NOT NULL DEFAULT 1,
  created_at          TIMESTAMPTZ DEFAULT NOW(),
  updated_at          TIMESTAMPTZ DEFAULT NOW(),
  created_by          UUID REFERENCES auth.users(id),
  approved_by         UUID REFERENCES auth.users(id)
);

-- Immutable audit log of all taxonomy changes
CREATE TABLE taxonomy_versions (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  change_type         TEXT NOT NULL CHECK (change_type IN (
                        'create', 'rename', 'merge', 'split',
                        'retire', 'description_update', 'signal_update'
                      )),
  affected_code       TEXT NOT NULL,         -- the code being changed
  old_values          JSONB NOT NULL,        -- snapshot of node before change
  new_values          JSONB NOT NULL,        -- snapshot of node after change
  related_codes       TEXT[],               -- codes involved in merge/split
  change_reason       TEXT,
  reclassification_scope TEXT CHECK (reclassification_scope IN (
                        'none', 'gap_flagged', 'all_in_l2', 'date_range', 'full'
                      )),
  reclassification_triggered_at TIMESTAMPTZ,
  entities_reclassified INTEGER DEFAULT 0,
  taxonomy_version_number INTEGER NOT NULL,  -- global sequence, incremented on every change
  changed_by          UUID REFERENCES auth.users(id),
  changed_at          TIMESTAMPTZ DEFAULT NOW()
);

-- Global version counter (single row)
CREATE TABLE taxonomy_meta (
  singleton           BOOLEAN PRIMARY KEY DEFAULT TRUE,
  current_version     INTEGER NOT NULL DEFAULT 1,
  last_changed_at     TIMESTAMPTZ DEFAULT NOW(),
  CONSTRAINT one_row CHECK (singleton = TRUE)
);

-- Trigger: auto-increment taxonomy_meta.current_version on any insert to taxonomy_versions
CREATE OR REPLACE FUNCTION increment_taxonomy_version()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE taxonomy_meta SET current_version = current_version + 1, last_changed_at = NOW();
  NEW.taxonomy_version_number := (SELECT current_version FROM taxonomy_meta);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_taxonomy_version_increment
  BEFORE INSERT ON taxonomy_versions
  FOR EACH ROW EXECUTE FUNCTION increment_taxonomy_version();

-- Gap events table (referenced in Part 2)
CREATE TABLE taxonomy_gap_events (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  entity_id           UUID NOT NULL REFERENCES kg_entities(id),
  signal_type         TEXT NOT NULL CHECK (signal_type IN (
                        'l3_miss', 'nonexistent_code', 'cluster', 'orphaned_concept'
                      )),
  suggested_code      TEXT,
  concept_name        TEXT,
  gap_reasoning       TEXT,
  associated_l2_domain TEXT,
  gap_cluster_id      UUID,
  resolved            BOOLEAN DEFAULT FALSE,
  resolved_by_code    TEXT REFERENCES taxonomy_nodes(code),
  resolved_at         TIMESTAMPTZ,
  detected_at         TIMESTAMPTZ DEFAULT NOW(),
  jurisdiction        TEXT NOT NULL DEFAULT 'FL'
);

-- Taxonomy proposals (from gap resolution workflow)
CREATE TABLE taxonomy_proposals (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  proposed_code       TEXT NOT NULL,
  parent_code         TEXT NOT NULL REFERENCES taxonomy_nodes(code),
  label               TEXT NOT NULL,
  description         TEXT NOT NULL,
  classification_signals JSONB NOT NULL,
  proposed_by         UUID REFERENCES auth.users(id),
  reviewed_by         UUID REFERENCES auth.users(id),
  status              TEXT NOT NULL DEFAULT 'proposed'
                        CHECK (status IN ('proposed', 'approved', 'rejected')),
  rejection_reason    TEXT,
  linked_gap_cluster_id UUID,
  proposed_at         TIMESTAMPTZ DEFAULT NOW(),
  reviewed_at         TIMESTAMPTZ
);
```

**Querying taxonomy at a specific version** (for audit/reproducibility):

```sql
-- Reconstruct the taxonomy as it existed at version N
-- (Not needed in hot path — used for audit reports and re-classification rollback)
SELECT tv.affected_code, tv.old_values
FROM taxonomy_versions tv
WHERE tv.taxonomy_version_number <= :target_version
ORDER BY tv.taxonomy_version_number DESC;
-- Then apply old_values snapshots in reverse to reconstruct prior state
```

For the common case — "what version of the taxonomy was used to classify entity X?" — store `taxonomy_version_number` on `kg_classification_log` rows. This is a single integer lookup.

### 3B. Backward Compatibility — Migration Logic

**Rename (code changes, concept stays the same)**:

```
RENAME MIGRATION PROTOCOL
Input: old_code → new_code (same concept, different identifier)
Example: "state-regulations.scope-of-practice" → "state-regulations.scope-practice"

Steps:
1. INSERT new node into taxonomy_nodes with new_code, copying all signals
2. UPDATE old node: status='deprecated', merged_into_code=new_code
3. INSERT into taxonomy_versions: change_type='rename', old_values={old_code}, new_values={new_code}
4. UPDATE kg_entity_domains SET domain_code=new_code WHERE domain_code=old_code
   -- This is the only DML migration needed; entity records themselves unchanged
5. UPDATE kg_classification_log: add column old_domain_code=old_code where applicable
   -- Audit trail preserved: both old and new codes visible on historical records
6. Set reclassification_scope='none' — rename does not change what the domain means,
   so no re-classification required

Audit trail behavior:
- All kg_classification_log rows retain their original code + a joined view
  can show the rename history via taxonomy_versions
- The old code is never deleted from taxonomy_versions; it is permanently archived
```

**Merge (two domains → one)**:

```
MERGE MIGRATION PROTOCOL
Input: code_A + code_B → code_merged (new combined domain)
Example: "telehealth.audio-only" + "telehealth.asynchronous" → "telehealth.non-synchronous"

Steps:
1. INSERT new merged node into taxonomy_nodes (code_merged)
2. UPDATE code_A: status='merged_into', merged_into_code=code_merged
3. UPDATE code_B: status='merged_into', merged_into_code=code_merged
4. INSERT into taxonomy_versions: change_type='merge', related_codes=[code_A, code_B]
5. For entities with domain_code=code_A OR domain_code=code_B:
   - UPDATE kg_entity_domains SET domain_code=code_merged
   - Confidence resolution on conflict: if entity had BOTH code_A and code_B (multi-domain),
     keep the HIGHER confidence value as the merged confidence.
     If the two confidences differ by >= 0.15, flag entity for HITL review
     (significant disagreement about what concept applied suggests the entity
     genuinely spans both pre-merge concepts in a way that merging obscures).
6. Set reclassification_scope='gap_flagged' — only re-run Stage 4 on entities
   where the merge conflict flag was set.

Backward compatibility:
- code_A and code_B permanently retained in taxonomy_nodes with status='merged_into'
- Historical classification logs remain readable: old code → merged code mapping
  accessible via taxonomy_versions
- Any external reference to code_A resolves to code_merged via a lookup function:
    SELECT COALESCE(merged_into_code, code) FROM taxonomy_nodes WHERE code = :input_code
```

**Split (one domain → two)**:

```
SPLIT MIGRATION PROTOCOL
Input: code_parent → code_child_A + code_child_B
Example: "state-regulations.licensure" →
         "state-regulations.licensure.physician" + "state-regulations.licensure.facility"

Steps:
1. INSERT code_child_A and code_child_B into taxonomy_nodes
   with split_from_code=code_parent
2. UPDATE code_parent: status='deprecated' (keep in tree for backward compat)
   Do NOT set merged_into_code (it didn't merge, it split)
3. INSERT into taxonomy_versions: change_type='split', related_codes=[code_child_A, code_child_B]
4. Determine reclassification scope:
   a. If total entities in code_parent < 500: scope='all_in_l2'
      Re-classify ALL entities that had code_parent
   b. If total entities >= 500: scope='date_range'
      Re-classify entities from the past 12 months (most likely still-current content)
      Queue older entities as low-priority background re-classification
5. Emit 'taxonomy/split.reclassify' event with entity_ids and new child codes
6. Stage 4 AI re-classification runs with updated taxonomy context.
   Prior reasoning from gap_reasoning stored at original classification reused
   as additional context in the re-classification prompt.

Entities that cannot be confidently re-classified to either child:
   - Classify to the deprecated parent code temporarily
   - Set taxonomy_gap_flag=true with signal_type='split_ambiguous'
   - Route to HITL queue for manual assignment
```

### 3C. Triggering Re-Classification

For each taxonomy change type, the scope of re-classification, pipeline stages to re-run, and expected volume:

| Change Type | Re-class Scope | Stages to Re-Run | Expected Volume |
|---|---|---|---|
| **Rename** | None (no conceptual change) | None | 0 entities |
| **Merge** | Merge-conflict entities only | Stage 4 only (AI decision) | 1–5% of merged entities |
| **Split** | All entities in deprecated parent (<500) OR last-12-months (>=500) | Stage 4 only | Varies: typically 50–500 for FL sources |
| **New domain added** | Gap-flagged entities in same L2 parent | Stage 4 only (with new taxonomy context) | All gap_flagged entities in that L2 area |
| **Domain retired** | All active entities with that code | Stage 4 only — assign to nearest active parent | All entities with retired code |
| **Signal update** (keywords/agency added) | None mandatory; optional low-priority batch | Stage 1-3a (deterministic only) | Optional; run in background |

Re-classification events are batched to avoid AI cost spikes:
- Maximum 50 entities per Inngest function invocation
- Fan out with `inngest.sendMany()` when scope exceeds 50 entities
- Re-classification runs at lower concurrency (limit: 5) to avoid displacing live entity classification

### 4B. Anomaly-Based Alert — Novelty Detection

**Alert name**: `ANOM_NEW_PAIR` — New Source-Domain Combination Detected

**Trigger condition**:
```
Entity is classified with confidence >= 0.75 to domain D,
AND source S has never previously been classified to domain D.
(Defined as: no row in source_domain_frequency where source_slug=S AND domain_code=D)
```

**Source-domain frequency matrix** (the operational data structure):

```sql
CREATE TABLE source_domain_frequency (
  source_slug         TEXT NOT NULL,
  domain_code         TEXT NOT NULL,
  jurisdiction        TEXT NOT NULL DEFAULT 'FL',
  first_seen_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_seen_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  occurrence_count    INTEGER NOT NULL DEFAULT 1,
  alert_sent          BOOLEAN DEFAULT FALSE,
  PRIMARY KEY (source_slug, domain_code, jurisdiction)
);

-- Update on each new classification (upsert)
INSERT INTO source_domain_frequency (source_slug, domain_code, jurisdiction, last_seen_at, occurrence_count)
VALUES ($1, $2, $3, NOW(), 1)
ON CONFLICT (source_slug, domain_code, jurisdiction)
DO UPDATE SET
  last_seen_at = NOW(),
  occurrence_count = source_domain_frequency.occurrence_count + 1;
```

**Detection logic** (in `persistClassification()` after matrix upsert):

```typescript
const wasNewPair = insertResult.rowCount === 1;  // 1 = first-ever occurrence (no conflict)
// If ON CONFLICT triggered, rowCount = 0 (existing pair, no alert)

if (wasNewPair && classification.confidence >= 0.75) {
  await inngest.send({
    name: "classification/anomaly.new_source_domain_pair",
    data: {
      entity_id,
      source_slug,
      domain_code: classification.domain_code,
      confidence: classification.confidence,
      reasoning: classification.reasoning,
    }
  });
}
```

**Alert format**:
```typescript
interface NewPairAlert {
  alert_type: 'ANOM_NEW_PAIR';
  entity_id: string;
  entity_title: string;
  source_slug: string;
  domain_code: string;           // the domain never before seen from this source
  classification_confidence: number;
  reasoning: string;             // AI's reasoning for the classification
  source_historical_domains: string[];  // domains this source has been seen in before
  jurisdiction: string;
  detected_at: string;
}
```

**Delivery**: Admin panel notification. If confidence >= 0.85 (high-confidence novel pair): also email to `reviewer` role. Novel high-confidence classifications are more likely to be genuine new regulatory territory than classification errors.

**Recommended response**:
- If source is `high_relevance` AND confidence >= 0.85: likely genuine; accept and monitor for follow-on entities
- If source is `mixed_relevance` AND confidence 0.75–0.84: manual HITL review recommended
- If same source produces 3+ new pairs in 7 days: audit source feed for scraping errors or source-scope change

**How the matrix is built initially**: On first deployment, run a backfill job that inserts one row per unique (source_slug, domain_code) pair from historical `kg_classification_log` entries. Set `alert_sent=true` on all backfill rows so they do not generate spurious alerts.

### 4C. Volume-Based Alert — Domain Spike Detection

**Alert name**: `VOL_SPIKE` — Unusual Classification Volume in Single Domain

**Detection logic**:

Rolling window comparison: current week vs. trailing 4-week average, alert when volume exceeds 2 standard deviations above the mean.

```sql
-- Run daily as a cron job (Inngest scheduled function)
WITH weekly_volumes AS (
  SELECT
    domain_code,
    DATE_TRUNC('week', created_at) AS week_start,
    COUNT(*)                        AS entity_count
  FROM kg_classification_log
  WHERE created_at > NOW() - INTERVAL '5 weeks'
    AND is_primary = true
  GROUP BY 1, 2
),
trailing_stats AS (
  SELECT
    domain_code,
    AVG(entity_count)    AS trailing_avg,
    STDDEV(entity_count) AS trailing_stddev
  FROM weekly_volumes
  WHERE week_start < DATE_TRUNC('week', NOW())   -- exclude current week
  GROUP BY domain_code
),
current_week AS (
  SELECT
    domain_code,
    COUNT(*) AS current_count
  FROM kg_classification_log
  WHERE created_at >= DATE_TRUNC('week', NOW())
    AND is_primary = true
  GROUP BY domain_code
)
SELECT
  cw.domain_code,
  cw.current_count,
  ts.trailing_avg,
  ts.trailing_stddev,
  CASE
    WHEN ts.trailing_stddev > 0
    THEN (cw.current_count - ts.trailing_avg) / ts.trailing_stddev
    ELSE NULL
  END AS z_score
FROM current_week cw
JOIN trailing_stats ts ON cw.domain_code = ts.domain_code
WHERE (cw.current_count - ts.trailing_avg) / NULLIF(ts.trailing_stddev, 0) >= 2.0
  AND cw.current_count >= 10    -- minimum absolute volume to avoid noise on small domains
ORDER BY z_score DESC NULLS LAST;
```

**Alert threshold**: z_score >= 2.0 AND current_count >= 10 entities this week.

**Distinguishing classification bug vs. genuine regulatory burst**:

The following secondary signals disambiguate:

| Signal | Suggests Bug | Suggests Genuine Burst |
|---|---|---|
| All spiking entities from a single source | Likely scraping error or source change | May be genuine if source is authoritative |
| Spiking entities span multiple sources | Unlikely to be a bug | Strong evidence of genuine activity |
| Average confidence during spike is LOW (< 0.65) | Likely misclassification cascade | Low confidence is ambiguous |
| Average confidence during spike is HIGH (>= 0.85) | Unlikely to be a bug | Strong evidence of genuine activity |
| Source feed shows unusual volume increase too | Correlates with source issue | Correlates with genuine activity |
| External check: does the domain align with known regulatory calendar? | No known activity → possible bug | Known rulemaking season → genuine |

```typescript
// Secondary signal check (runs when spike alert fires)
async function classifySpikeCause(domainCode: string, weekStart: Date): Promise<'likely_bug' | 'likely_genuine' | 'ambiguous'> {
  const [sourceCount, avgConfidence] = await Promise.all([
    countDistinctSources(domainCode, weekStart),
    getAvgConfidence(domainCode, weekStart),
  ]);

  if (sourceCount === 1 && avgConfidence < 0.65) return 'likely_bug';
  if (sourceCount >= 3 && avgConfidence >= 0.80) return 'likely_genuine';
  return 'ambiguous';
}
```

**Alert format**:
```typescript
interface VolumeSpikeAlert {
  alert_type: 'VOL_SPIKE';
  domain_code: string;
  current_week_count: number;
  trailing_4week_avg: number;
  trailing_stddev: number;
  z_score: number;
  likely_cause: 'likely_bug' | 'likely_genuine' | 'ambiguous';
  distinct_source_count: number;
  avg_confidence_in_spike: number;
  example_entity_ids: string[];     // first 5 entities in the spike
  week_start: string;
}
```

**Delivery**: Admin panel notification. If `likely_cause = 'likely_bug'`: also email to admin role with "Action required" subject line.

**Recommended response**:
- `likely_bug`: Audit the source feed. Check Stage 1/3a keyword rules for the spiking domain. Pause Stage 4 AI classification for that domain if confidence pattern is anomalous.
- `likely_genuine`: Proceed normally. Consider creating a digest alert for practice owners about the regulatory burst.
- `ambiguous`: Admin review within 24 hours. Compare with external regulatory calendar.

### 4D. Conflict-Based Alert — Cross-Entity Classification Conflict

**Alert name**: `CONF_CONFLICT` — Adjacent Entities Classified in Different L1 Domains

**Trigger condition**:
```
New entity E_new is classified to primary L1 domain D_new.
A related entity E_existing is found in the knowledge graph (same source, adjacent citation).
E_existing has primary L1 domain D_existing.
D_new != D_existing.
```

**Defining "closely related" operationally**:

Two entities are closely related if ANY of the following are true:
1. Same source slug AND citation strings are adjacent (e.g., "21 CFR 1306.04" and "21 CFR 1306.05" — same part, consecutive sections)
2. Same source slug AND same publication date AND both classified after the same pipeline run
3. Same federal document (FR document number or state notice number) AND different sections

Citation adjacency check:
```typescript
function areAdjacentCitations(cite1: string, cite2: string): boolean {
  // Parse CFR format: "21 CFR 1306.04" → { title: 21, part: 1306, section: 4 }
  const p1 = parseCitation(cite1);
  const p2 = parseCitation(cite2);
  if (!p1 || !p2) return false;

  // Same title + same part + sections differ by 1
  return p1.title === p2.title
    && p1.part === p2.part
    && Math.abs((p1.section ?? 0) - (p2.section ?? 0)) <= 2;
}
```

**Detection logic** (runs after each new classification, in `persistClassification()`):

```typescript
// Check for conflicting related entities
const relatedEntities = await supabase
  .from("kg_entities")
  .select("id, citation, primary_domain_code")
  .eq("source_slug", source_slug)
  .neq("id", entity_id)
  .order("created_at", { ascending: false })
  .limit(20);

for (const related of relatedEntities.data ?? []) {
  const isRelated = areAdjacentCitations(newEntity.citation, related.citation)
    || (newEntity.publication_date === related.publication_date);

  if (isRelated) {
    const newL1 = newEntity.primary_domain_code.split(".")[0];
    const existingL1 = related.primary_domain_code.split(".")[0];

    if (newL1 !== existingL1) {
      await inngest.send({
        name: "classification/conflict.detected",
        data: { entity_id_new: entity_id, entity_id_existing: related.id, ... }
      });
    }
  }
}
```

**Alert format**:
```typescript
interface ConflictAlert {
  alert_type: 'CONF_CONFLICT';
  entity_id_new: string;
  entity_id_existing: string;
  new_entity_title: string;
  existing_entity_title: string;
  new_entity_domain: string;        // e.g., "telehealth.prescribing"
  existing_entity_domain: string;   // e.g., "controlled-substances.prescribing"
  new_entity_confidence: number;
  existing_entity_confidence: number;
  relationship_type: 'adjacent_citation' | 'same_publication_date' | 'same_document';
  citation_new: string;
  citation_existing: string;
  source_slug: string;
  jurisdiction: string;
  detected_at: string;
}
```

**Delivery**: Admin panel notification. If both entities have confidence >= 0.85 (high-confidence conflict, possibly the more serious kind): also email to reviewer role.

**Note on expected frequency**: Genuine cross-domain conflicts are common and often correct. A telehealth section and a controlled substances section of the same DEA document should be classified in different L1 domains. The alert's purpose is to surface conflicts for human verification, not to assert that one classification is wrong. Response: the reviewer inspects both entities and either confirms both are correct (no action) or corrects one.

## Part 5: Extended Monitoring Metrics

P2-S5 defined 10 core metrics. The following 5 new metrics address gaps in coverage for non-federal sources, real-time single-entity mode, and taxonomy evolution tracking.

### Metric 11: State Source Classification Rate

**Purpose**: Tracks how effectively the pipeline classifies entities from state-level sources (FL boards, FL Administrative Register, FL legislature). State source classification relies more heavily on Stage 4 AI than federal sources — this metric surfaces degradation in state-specific rule coverage.

**Calculation SQL**:
```sql
SELECT
  DATE(created_at)           AS date,
  source_slug,
  jurisdiction,
  COUNT(*)                   AS total_entities,
  COUNT(*) FILTER (WHERE stage IN ('structural', 'metadata') AND s3_tier = 'relevant')
                             AS classified_deterministically,
  COUNT(*) FILTER (WHERE stage = 'ai')
                             AS required_ai,
  COUNT(*) FILTER (WHERE s3_tier = 'relevant')
                             AS fully_classified,
  ROUND(
    COUNT(*) FILTER (WHERE s3_tier = 'relevant') * 100.0 / NULLIF(COUNT(*), 0),
    1
  )                          AS classification_rate_pct,
  AVG(confidence) FILTER (WHERE s3_tier = 'relevant')
                             AS avg_confidence_classified
FROM kg_classification_log
WHERE jurisdiction != 'federal'
  AND created_at > NOW() - INTERVAL '30 days'
GROUP BY 1, 2, 3
ORDER BY 1 DESC, 5 ASC;   -- show lowest-classified sources first
```

**Alert threshold**: Any non-federal source with `classification_rate_pct < 60%` over a 7-day rolling window.

**Dashboard display**: Table widget "State Source Coverage" in admin Classification dashboard. One row per source. Red highlight on any source below threshold.

### Metric 12: Single-Entity Latency P95 (Real-Time Mode)

**Purpose**: Tracks end-to-end latency for live entity classification (from `monitor/entity.new` event fire to `classification_status` being set). Distinct from the batch latency P95 in P2-S5's metric #7 — real-time mode has a stricter user-facing SLA.

**Calculation SQL**:
```sql
SELECT
  DATE(classified_at)                          AS date,
  source_slug,
  PERCENTILE_CONT(0.50) WITHIN GROUP (
    ORDER BY EXTRACT(EPOCH FROM (classified_at - ingested_at))
  )                                            AS p50_seconds,
  PERCENTILE_CONT(0.95) WITHIN GROUP (
    ORDER BY EXTRACT(EPOCH FROM (classified_at - ingested_at))
  )                                            AS p95_seconds,
  PERCENTILE_CONT(0.99) WITHIN GROUP (
    ORDER BY EXTRACT(EPOCH FROM (classified_at - ingested_at))
  )                                            AS p99_seconds,
  COUNT(*)                                     AS sample_count,
  COUNT(*) FILTER (WHERE EXTRACT(EPOCH FROM (classified_at - ingested_at)) > 480)
                                               AS exceeded_budget_count
FROM kg_entities
WHERE ingested_at > NOW() - INTERVAL '7 days'
  AND classification_mode = 'realtime'   -- column added to kg_entities to distinguish
GROUP BY 1, 2
ORDER BY 1 DESC, p95_seconds DESC;
```

**Alert threshold**: P95 latency > 480 seconds (8 minutes, the target latency budget from Part 1A) on any day with >= 10 entities.

**Dashboard display**: Line chart "Real-Time Classification Latency" with P50/P95/P99 lines. Budget line at 480s shown as a horizontal reference.

### Metric 13: Taxonomy Gap Rate

**Purpose**: Tracks the rate at which new entities arrive that cannot be fully classified to L3+ within the existing taxonomy. A rising gap rate signals that the taxonomy is falling behind the regulatory landscape — either because new practice areas are emerging or because Cedar is monitoring new source types.

**Calculation SQL**:
```sql
SELECT
  DATE_TRUNC('week', detected_at) AS week,
  signal_type,
  jurisdiction,
  COUNT(*)                         AS gap_events,
  COUNT(DISTINCT entity_id)        AS unique_entities_affected,
  COUNT(DISTINCT associated_l2_domain)
                                   AS l2_domains_with_gaps,
  COUNT(*) FILTER (WHERE resolved = TRUE)
                                   AS resolved_count,
  ROUND(
    COUNT(*) FILTER (WHERE resolved = TRUE) * 100.0 / NULLIF(COUNT(*), 0),
    1
  )                                AS resolution_rate_pct
FROM taxonomy_gap_events
WHERE detected_at > NOW() - INTERVAL '90 days'
GROUP BY 1, 2, 3
ORDER BY 1 DESC, 4 DESC;
```

**Alert threshold**: `unique_entities_affected > 20` in any single week, OR `resolution_rate_pct < 50%` over a 30-day window (gaps being detected faster than they are resolved).

**Dashboard display**: Stacked bar chart "Taxonomy Gap Events by Week" with signal_type as the stacking dimension. Secondary line: resolution rate. Admin panel only.

### Metric 14: Board Minutes and Legislative Document Coverage

**Purpose**: Board meeting minutes and legislative documents (FL House/Senate bills, committee reports) are inherently unstructured and have the lowest deterministic classification coverage of any source type. This metric specifically tracks how many such documents are reaching Stage 4 AI (vs. being classified deterministically or falling through to HITL), and the resulting confidence distribution. It quantifies the cost and accuracy of Cedar's most expensive source-type to classify.

**Calculation SQL**:
```sql
SELECT
  DATE(kcl.created_at)          AS date,
  ks.source_type,               -- 'board_minutes' | 'legislative_bill' | 'committee_report'
  COUNT(*)                      AS total_classified,
  COUNT(*) FILTER (WHERE kcl.stage = 'ai')
                                AS reached_stage_4,
  ROUND(
    COUNT(*) FILTER (WHERE kcl.stage = 'ai') * 100.0 / NULLIF(COUNT(*), 0),
    1
  )                             AS pct_requiring_ai,
  AVG(kcl.confidence) FILTER (WHERE kcl.stage = 'ai')
                                AS avg_ai_confidence,
  SUM(kcl.api_cost_usd)         AS total_cost_usd,
  COUNT(*) FILTER (WHERE kcl.confidence < 0.60)
                                AS low_confidence_count,
  COUNT(*) FILTER (WHERE kcl.taxonomy_gap_flag = TRUE)
                                AS gap_flagged_count
FROM kg_classification_log kcl
JOIN kg_entities ke ON ke.id = kcl.entity_id
JOIN monitoring_sources ks ON ks.slug = ke.source_slug
WHERE ks.source_type IN ('board_minutes', 'legislative_bill', 'committee_report')
  AND kcl.created_at > NOW() - INTERVAL '30 days'
GROUP BY 1, 2
ORDER BY 1 DESC, 3 DESC;
```

**Alert threshold**: `avg_ai_confidence < 0.65` over a 14-day window for any board minutes source, OR `pct_requiring_ai > 90%` (indicating near-zero deterministic coverage for that source — likely the keyword/structural rules need expansion).

**Dashboard display**: Table "Board & Legislative Source Performance" showing AI usage rate and average confidence per source type. Color-coded: green (avg confidence >= 0.75), yellow (0.65–0.74), red (< 0.65).

### Metric 15: Taxonomy Version Velocity and Re-Classification Debt

**Purpose**: Tracks how frequently the taxonomy is changing and whether the re-classification work triggered by those changes is being completed or accumulating. A growing re-classification backlog means entities in the Library may have stale domain assignments.

**Calculation SQL**:
```sql
-- Part A: Taxonomy change velocity (last 90 days)
SELECT
  DATE_TRUNC('month', changed_at)   AS month,
  change_type,
  COUNT(*)                          AS changes,
  SUM(entities_reclassified)        AS entities_reclassified,
  AVG(
    EXTRACT(EPOCH FROM (reclassification_triggered_at - changed_at)) / 3600
  )                                 AS avg_hours_to_reclassify
FROM taxonomy_versions
WHERE changed_at > NOW() - INTERVAL '90 days'
GROUP BY 1, 2
ORDER BY 1 DESC;

-- Part B: Re-classification debt (pending entities)
SELECT
  COUNT(*)                          AS total_gap_flagged,
  COUNT(*) FILTER (WHERE resolved = FALSE)
                                    AS unresolved_gap_entities,
  MIN(detected_at)                  AS oldest_unresolved,
  COUNT(DISTINCT associated_l2_domain) FILTER (WHERE resolved = FALSE)
                                    AS domains_with_debt
FROM taxonomy_gap_events;
```

**Alert threshold**: `unresolved_gap_entities > 200` OR `oldest_unresolved < NOW() - INTERVAL '30 days'` (30+ day old unresolved gap entities indicate a taxonomy that is not being maintained).

**Dashboard display**: Two-panel widget "Taxonomy Health". Left: bar chart of taxonomy changes by type per month. Right: gauge showing unresolved gap entity count vs. 200-entity threshold. Admin panel only.

## Summary

This document defines Cedar's operational steady-state: what happens to each entity as it arrives, how the taxonomy stays current as the regulatory landscape evolves, and how the system surfaces problems that require human intervention.

Key design decisions:

1. **Short-circuit aggressively**. For FL Board sources with deterministic structural matches, most entities will classify in under 200ms. The 8-minute budget is reserved for entities genuinely requiring AI judgment.

2. **Source authority compensates for classification uncertainty**. The relevance gate and routing rules explicitly encode the insight that a low-confidence classification from the FL Board of Medicine is still more useful to practice owners than withholding it — the source's authority justifies early visibility.

3. **Taxonomy gaps are a first-class operational concern, not edge cases**. The gap event pipeline, proposal workflow, and Metric 13 treat taxonomy maintenance as an ongoing operational discipline, not an occasional cleanup task.

4. **Alerting is layered by severity and role**. Practice owners never see internal classification alerts. Admin and reviewer roles receive graduated alerts (panel notification → email → "action required" email) calibrated to avoid fatigue.

5. **Backward compatibility is non-negotiable**. Every taxonomy change preserves the complete history of prior classifications through immutable `taxonomy_versions` rows. Old domain codes resolve to their current equivalents via a single lookup.
