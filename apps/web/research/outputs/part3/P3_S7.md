## Overview

This session closes the Cedar classification framework by covering three cross-cutting operational concerns:

1. **Audit trail schema extensions** — extending P2-S4's `kg_classification_log` to cover authority level assignments, non-federal source classifications, and jurisdiction-scoped queries
2. **Complete metrics suite** — reconciling P2-S5's 10 metrics and P3-S6's 5 additional metrics into a single canonical list, adding 5 further operational metrics, and defining the consolidated admin dashboard layout
3. **Bulk re-classification pipeline** — scope determination, stage selection, conflict resolution, rate limiting, Inngest architecture, and rollback

Upstream design anchors:
- **P2-S4** defined `kg_classification_log` with `ai_reasoning`, `confidence_scores`, `cross_domains`, `corrected_from`, `tokens_used`, `prompt_version`, `model_version`. That schema is extended here, not redesigned.
- **P2-S5** defined 10 monitoring metrics with SQL, alert thresholds, and dashboard views.
- **P3-S4** defined the `authority_level` enum (8 values), `authority_level_hierarchy` table, and transition lifecycle (proposed → active → superseded).
- **P3-S6** defined taxonomy evolution schemas (`taxonomy_nodes`, `taxonomy_versions`, `taxonomy_gap_events`, `source_domain_frequency`), alerting rules (`CONF_LOW`, `ANOM_NEW_PAIR`, `VOL_SPIKE`, `CONF_CONFLICT`), and 5 extended metrics (Metrics 11–15). It also defined the routing rules R1–R4, with R4 applying at `high_relevance AND 0.65 <= confidence < 0.85`.

## Section 1: Audit Trail Schema Extensions

### 1A. Authority Level Audit

P2-S4's `kg_classification_log` captures domain classification events. Authority level assignment is a parallel event that deserves equal audit rigor. The following DDL extends `kg_classification_log` and adds a dedicated authority level transition log.

#### 1A-1. Extensions to `kg_classification_log`

```sql
-- Extend kg_classification_log to capture authority level assignment alongside domain classification.
-- These columns are nullable because Stage 1–3 entries (deterministic stages) may not have
-- authority level information yet; Stage 4 (AI) entries should always populate them.

ALTER TABLE public.kg_classification_log
  ADD COLUMN IF NOT EXISTS authority_level         public.authority_level,
  ADD COLUMN IF NOT EXISTS authority_level_source  TEXT
    CHECK (authority_level_source IN (
      'structural_rule',    -- Stage 1: deterministic FR/eCFR type → authority level rule
      'metadata_rule',      -- Stage 2: agency/doc-type mapping
      'ai_classification',  -- Stage 4: AI assigned authority level
      'hitl_correction',    -- Human reviewer overrode authority level
      'lifecycle_transition' -- System updated (proposed → active, etc.)
    )),
  ADD COLUMN IF NOT EXISTS authority_level_confidence NUMERIC(4,3)
    CHECK (authority_level_confidence BETWEEN 0 AND 1),
  ADD COLUMN IF NOT EXISTS authority_level_reasoning TEXT,
  ADD COLUMN IF NOT EXISTS entity_status            TEXT
    CHECK (entity_status IN (
      'active', 'proposed', 'pending_enforcement', 'superseded', 'expired'
    )),
  ADD COLUMN IF NOT EXISTS taxonomy_version_number  INTEGER,  -- snapshot of taxonomy at classification time
  ADD COLUMN IF NOT EXISTS jurisdiction             TEXT NOT NULL DEFAULT 'FL';

-- Index: authority level queries scoped by jurisdiction
CREATE INDEX IF NOT EXISTS idx_classification_log_authority_jurisdiction
  ON public.kg_classification_log (authority_level, jurisdiction)
  WHERE authority_level IS NOT NULL;

-- Index: authority level transitions over time (for transition query in 1C)
CREATE INDEX IF NOT EXISTS idx_classification_log_authority_created
  ON public.kg_classification_log (authority_level, created_at DESC)
  WHERE authority_level_source = 'lifecycle_transition';

-- Index: jurisdiction + confidence for the cross-comparison query in 1C
CREATE INDEX IF NOT EXISTS idx_classification_log_jurisdiction_confidence
  ON public.kg_classification_log (jurisdiction, (confidence_scores->0->>'confidence')::float)
  WHERE stage = 4;
```

#### 1A-2. Authority Level Transition Log

Domain classifications are tracked in `kg_classification_log`. Authority level *transitions* (proposed → pending_enforcement → active → superseded) are distinct events that need their own immutable log. This log is separate from domain classification to preserve orthogonality.

```sql
-- Immutable log of every authority level state change on a kg_entity.
-- Append-only: never UPDATE or DELETE rows in this table.
-- Backed by the same append-only guarantee as kg_classification_log.

CREATE TABLE IF NOT EXISTS public.authority_level_transitions (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  entity_id             UUID NOT NULL REFERENCES public.kg_entities(id),
  from_authority_level  public.authority_level,     -- NULL for initial assignment
  to_authority_level    public.authority_level NOT NULL,
  from_status           TEXT,                       -- NULL for initial assignment
  to_status             TEXT NOT NULL,
  transition_type       TEXT NOT NULL CHECK (transition_type IN (
    'initial_assignment',  -- first assignment when entity ingested
    'proposed_to_final',   -- proposed rule/bill enacted/finalized
    'final_to_active',     -- pending_enforcement → active (effective date reached)
    'superseded',          -- replaced by a newer entity
    'expired',             -- emergency rule lapsed
    'hitl_correction',     -- human reviewer corrected authority level
    'authority_upgrade',   -- e.g., DEA scheduling notice reclassified from sub_reg to federal_reg
    'status_correction'    -- typo/error correction (requires admin role)
  )),
  trigger_source        TEXT,      -- event or job that triggered the transition
  prior_entity_id       UUID REFERENCES public.kg_entities(id), -- for superseded transitions
  superseded_by_id      UUID REFERENCES public.kg_entities(id), -- for superseded transitions
  effective_date        DATE,      -- populated when transition_type = 'final_to_active'
  reviewer_id           UUID REFERENCES auth.users(id),  -- populated for hitl_correction
  notes                 TEXT,
  jurisdiction          TEXT NOT NULL DEFAULT 'FL',
  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Index: time-scoped transition queries (used by Metric 17 and 1C jurisdiction queries)
CREATE INDEX IF NOT EXISTS idx_authority_transitions_created
  ON public.authority_level_transitions (created_at DESC, transition_type);

CREATE INDEX IF NOT EXISTS idx_authority_transitions_entity
  ON public.authority_level_transitions (entity_id, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_authority_transitions_jurisdiction
  ON public.authority_level_transitions (jurisdiction, to_authority_level, created_at DESC);
```

#### 1A-3. Writing Authority Level Audit Entries

Every time an entity's authority level is assigned or transitions, two writes occur:
1. A row is inserted into `kg_classification_log` with `authority_level_source = 'lifecycle_transition'` (or the appropriate source).
2. A row is inserted into `authority_level_transitions` with the full transition context.

This dual-write ensures that domain classification history and authority level history are both queryable from the same table (`kg_classification_log`) while preserving a dedicated, clean transition log for authority-level-specific reporting.

### 1B. Non-Federal Source Audit

#### 1B-1. Board Minutes — Low-Confidence Audit Trail

Florida board meeting minutes have the lowest classification confidence profile of any source type (~65–70% L2 accuracy per P2-S5's error analysis). When classifying board minutes, the audit entry must capture the full disambiguation path.

```sql
-- Additional columns on kg_classification_log for non-federal source audit detail.
-- These are populated only when source_type IN ('board_minutes', 'board_order', 'state_agency_guidance').

ALTER TABLE public.kg_classification_log
  ADD COLUMN IF NOT EXISTS document_binding_signal TEXT
    CHECK (document_binding_signal IN (
      'title_pattern_match',  -- matched "Final Order", "Consent Agreement", etc.
      'case_number_present',  -- DOH-XX-XXXX-MQA pattern detected
      'parties_named',        -- named respondent(s) detected
      'doah_reference',       -- Division of Administrative Hearings case number
      'ai_disambiguated',     -- none of the above; AI classified binding vs non-binding
      'no_signal'             -- no disambiguation signals found
    )),
  ADD COLUMN IF NOT EXISTS binding_determination   BOOLEAN,  -- true = state_board_rule, false = sub_regulatory
  ADD COLUMN IF NOT EXISTS bill_lifecycle_stage    TEXT
    CHECK (bill_lifecycle_stage IN (
      'introduced', 'committee', 'passed_one_chamber',
      'passed_both_chambers', 'enrolled', 'signed', 'became_law_without_signature', 'vetoed'
    ));
  -- bill_lifecycle_stage: populated only when source_type = 'legislative_bill'
  -- Enables tracking how a bill's classification evolves as its status changes.
```

When the pipeline processes a board document:
- If a title pattern match fires (`document_binding_signal = 'title_pattern_match'`), `authority_level_confidence` is set to 0.95.
- If only `ai_disambiguated`, `authority_level_confidence` is set to the AI's confidence for the binding determination, capped at 0.85.
- If `no_signal`, `authority_level_confidence` defaults to 0.60 and `review_required = true`.

#### 1B-2. Legislative Bill Classification — Lifecycle Tracking

Florida legislative bills require audit entries at each lifecycle stage because classification changes as the bill progresses. The bill starts as `state_statute + status = 'proposed'` and transitions to `state_statute + status = 'active'` if enacted.

Each time Cedar detects a bill status change, a new row is inserted into `kg_classification_log` (not an update to the prior row) and a new row is inserted into `authority_level_transitions`. The prior row is retained unchanged — the immutable audit trail shows exactly when Cedar detected each stage transition and what classification was active at each point.

The `bill_lifecycle_stage` column on `kg_classification_log` enables this query: "what was Cedar's classification of HB 1234 when the bill was in committee?" — a question that may be relevant when a practice disputes whether they had adequate notice at a specific lifecycle stage.

### 1C. Jurisdiction-Aware Audit Queries

All three queries below are production-ready SQL against the extended `kg_classification_log` schema defined in P2-S4 and this session.

#### Query C1: FL Board Rule Classifications with Confidence < 0.70

```sql
-- All FL board rule classifications with confidence below 0.70.
-- Used for: weekly HITL review targeting, accuracy monitoring for board sources.
-- Performance: covered by idx_classification_log_authority_jurisdiction
--   + idx_classification_log_jurisdiction_confidence

SELECT
  kcl.entity_id,
  kcl.created_at                                              AS classified_at,
  kcl.authority_level,
  kcl.authority_level_source,
  kcl.authority_level_confidence,
  (kcl.confidence_scores -> 0 ->> 'domain_code')             AS primary_domain,
  (kcl.confidence_scores -> 0 ->> 'confidence')::NUMERIC      AS domain_confidence,
  kcl.document_binding_signal,
  kcl.binding_determination,
  kcl.authority_level_reasoning,
  kcl.ai_reasoning ->> 'chain_of_thought'                    AS reasoning_summary,
  ke.title                                                    AS entity_title,
  ke.source_slug
FROM public.kg_classification_log kcl
JOIN public.kg_entities ke
  ON ke.id = kcl.entity_id
WHERE kcl.jurisdiction = 'FL'
  AND kcl.authority_level = 'state_board_rule'
  AND (kcl.confidence_scores -> 0 ->> 'confidence')::NUMERIC < 0.70
  AND kcl.stage = 4
ORDER BY kcl.created_at DESC;
```

#### Query C2: Authority Level Transitions in the Last 30 Days

```sql
-- All authority level transitions in the last 30 days, with entity context.
-- Used for: compliance monitoring (track proposed→active transitions),
--   alerting dashboard, proposed-rule-to-final change feed.

SELECT
  alt.transition_type,
  alt.from_status,
  alt.to_status,
  alt.from_authority_level,
  alt.to_authority_level,
  alt.effective_date,
  alt.created_at                              AS transition_detected_at,
  alt.jurisdiction,
  ke.title                                   AS entity_title,
  ke.source_slug,
  alt.trigger_source,
  alt.reviewer_id,
  alt.notes,
  -- Link to the prior entity when superseded
  prior_ke.title                             AS superseded_entity_title
FROM public.authority_level_transitions alt
JOIN public.kg_entities ke
  ON ke.id = alt.entity_id
LEFT JOIN public.kg_entities prior_ke
  ON prior_ke.id = alt.prior_entity_id
WHERE alt.created_at > NOW() - INTERVAL '30 days'
ORDER BY alt.created_at DESC;
```

#### Query C3: Classification Accuracy Comparison — Federal vs. FL Sources

```sql
-- Compare classification accuracy (correction rate as proxy) between
-- federal sources and FL state sources, broken down by source_slug.
-- Used for: calibration monitoring, identifying sources needing rule coverage expansion.

WITH all_classifications AS (
  SELECT
    kcl.entity_id,
    kcl.jurisdiction,
    ke.source_slug,
    CASE
      WHEN ke.source_slug IN (
        'ecfr', 'federal-register', 'openfda', 'cms-transmittals'
      ) THEN 'federal'
      ELSE 'state'
    END                                               AS source_category,
    kcl.stage,
    (kcl.confidence_scores -> 0 ->> 'confidence')::NUMERIC AS domain_confidence
  FROM public.kg_classification_log kcl
  JOIN public.kg_entities ke ON ke.id = kcl.entity_id
  WHERE kcl.created_at > NOW() - INTERVAL '90 days'
    AND kcl.stage = 4
),
corrections AS (
  SELECT
    orig.entity_id,
    COUNT(*) AS correction_count
  FROM public.kg_classification_log orig
  JOIN public.kg_classification_log corr
    ON corr.corrected_from = orig.id
  WHERE orig.stage = 4
    AND corr.created_at > NOW() - INTERVAL '90 days'
  GROUP BY orig.entity_id
)
SELECT
  ac.source_category,
  ac.jurisdiction,
  COUNT(DISTINCT ac.entity_id)                                AS total_classified,
  AVG(ac.domain_confidence)                                   AS avg_confidence,
  COUNT(DISTINCT c.entity_id)                                 AS entities_corrected,
  ROUND(
    COUNT(DISTINCT c.entity_id) * 100.0 / NULLIF(COUNT(DISTINCT ac.entity_id), 0),
    2
  )                                                           AS correction_rate_pct,
  ROUND(
    (1 - COUNT(DISTINCT c.entity_id) * 1.0 / NULLIF(COUNT(DISTINCT ac.entity_id), 0)) * 100,
    2
  )                                                           AS estimated_accuracy_pct
FROM all_classifications ac
LEFT JOIN corrections c ON c.entity_id = ac.entity_id
GROUP BY ac.source_category, ac.jurisdiction
ORDER BY ac.source_category, ac.jurisdiction;
```

#### Metric 16: Authority Level Distribution

**Purpose**: Tracks the percentage of entities at each authority level. A healthy corpus should have a distribution that matches the known regulatory landscape (most entities are `federal_regulation` or `sub_regulatory_guidance`; very few are `federal_statute`). Unexpected shifts signal either a new source type being ingested or a systematic misclassification.

**Calculation SQL**:
```sql
SELECT
  authority_level,
  jurisdiction,
  COUNT(*)                                                      AS entity_count,
  ROUND(
    COUNT(*) * 100.0 / NULLIF(SUM(COUNT(*)) OVER (PARTITION BY jurisdiction), 0),
    1
  )                                                             AS pct_of_jurisdiction,
  AVG(authority_level_confidence)                               AS avg_assignment_confidence,
  COUNT(*) FILTER (WHERE entity_status = 'proposed')            AS proposed_count,
  COUNT(*) FILTER (WHERE entity_status = 'pending_enforcement') AS pending_enforcement_count,
  COUNT(*) FILTER (WHERE entity_status = 'active')              AS active_count
FROM public.kg_entities ke
JOIN public.authority_level_transitions alt
  ON alt.entity_id = ke.id
  AND alt.transition_type = 'initial_assignment'   -- one row per entity for current level
WHERE ke.created_at > NOW() - INTERVAL '30 days'   -- rolling window; remove for full corpus
GROUP BY authority_level, jurisdiction
ORDER BY jurisdiction, entity_count DESC;
```

**Alert threshold**: Any `authority_level` at > 25% of total entities in a jurisdiction when it was historically < 10% (suggests systematic misclassification). Or: `professional_standard` exceeds 5% of FL corpus (FL board scraping should not be producing professional standards at volume).

**Dashboard display**: Stacked bar chart by jurisdiction. One bar per jurisdiction, segments colored by `authority_level` using the badge color palette from P3-S4 Section 3C.

#### Metric 17: Authority Level Transition Volume

**Purpose**: Tracks how many authority level transitions occur per time period. A healthy system sees moderate proposed→final transitions (matching regulatory cycle). Sudden spikes in `superseded` or `expired` transitions can indicate bulk source updates or ingestion errors.

**Calculation SQL**:
```sql
SELECT
  DATE_TRUNC('week', alt.created_at)  AS week,
  alt.transition_type,
  alt.jurisdiction,
  COUNT(*)                            AS transition_count,
  COUNT(DISTINCT alt.entity_id)       AS distinct_entities
FROM public.authority_level_transitions alt
WHERE alt.created_at > NOW() - INTERVAL '12 weeks'
GROUP BY 1, 2, 3
ORDER BY 1 DESC, 4 DESC;
```

**Alert threshold**: More than 50 `superseded` transitions in a single week from any single source (may indicate a bulk ingestion error creating duplicate entity chains).

**Dashboard display**: Line chart with one series per `transition_type`. Secondary table below showing the top 5 sources by transition volume that week.

#### Metric 18: Jurisdiction Coverage

**Purpose**: Tracks the percentage of entities classified (reached a final classification status) by jurisdiction. Ensures state sources are being classified at a comparable rate to federal sources. A low FL classification rate suggests keyword coverage or Stage 4 prompt coverage gaps for state-specific regulatory language.

**Calculation SQL**:
```sql
SELECT
  ke.jurisdiction,
  COUNT(*)                                                               AS total_entities,
  COUNT(*) FILTER (WHERE ked.classification_status IN (
    'accepted', 'accepted_with_flag', 'accepted_source_override'
  ))                                                                     AS fully_classified,
  COUNT(*) FILTER (WHERE ked.classification_status = 'pending_review')  AS pending_review,
  COUNT(*) FILTER (WHERE ked.library_visible = TRUE)                    AS library_visible,
  ROUND(
    COUNT(*) FILTER (WHERE ked.classification_status IN (
      'accepted', 'accepted_with_flag', 'accepted_source_override'
    )) * 100.0 / NULLIF(COUNT(*), 0),
    1
  )                                                                     AS classification_rate_pct,
  AVG(
    (kcl.confidence_scores -> 0 ->> 'confidence')::NUMERIC
  ) FILTER (WHERE kcl.stage = 4)                                        AS avg_ai_confidence
FROM public.kg_entities ke
JOIN public.kg_entity_domains ked ON ked.entity_id = ke.id
LEFT JOIN public.kg_classification_log kcl ON kcl.entity_id = ke.id AND kcl.stage = 4
WHERE ke.created_at > NOW() - INTERVAL '30 days'
GROUP BY ke.jurisdiction
ORDER BY classification_rate_pct ASC;
```

**Alert threshold**: Any jurisdiction with `classification_rate_pct < 70%` over a 7-day rolling window. (FL is expected to be lower than federal at launch due to sparser rule coverage — acceptable floor is 70%, rising to 80% after keyword expansion.)

**Dashboard display**: Progress bar table. One row per jurisdiction with: entity count, classification rate %, avg AI confidence. Color scale: red < 70%, yellow 70–80%, green > 80%.

#### Metric 19: Source Type Accuracy Comparison

**Purpose**: Breaks down classification correction rate by source type (eCFR, Federal Register, FL FAC, board minutes, etc.) to identify which source types have the poorest classification accuracy. This metric is the source-type equivalent of P2-S5's M9 (error rate by domain).

**Calculation SQL**:
```sql
WITH source_type_classifications AS (
  SELECT
    ke.source_slug,
    -- Derive source type from source_slug via a simple mapping
    CASE
      WHEN ke.source_slug = 'ecfr'               THEN 'eCFR'
      WHEN ke.source_slug = 'federal-register'   THEN 'Federal Register'
      WHEN ke.source_slug = 'openfda'            THEN 'openFDA'
      WHEN ke.source_slug = 'cms-transmittals'   THEN 'CMS Transmittals'
      WHEN ke.source_slug LIKE 'fl-board-%'      THEN 'FL Board Documents'
      WHEN ke.source_slug = 'fl-administrative-register' THEN 'FL Admin Register'
      WHEN ke.source_slug = 'fl-legislature-bills'       THEN 'FL Legislature Bills'
      ELSE 'Other'
    END                                           AS source_type,
    kcl.entity_id,
    kcl.stage,
    kcl.id                                        AS log_id
  FROM public.kg_classification_log kcl
  JOIN public.kg_entities ke ON ke.id = kcl.entity_id
  WHERE kcl.stage = 4
    AND kcl.created_at > NOW() - INTERVAL '90 days'
),
correction_counts AS (
  SELECT
    orig.entity_id,
    COUNT(*) AS corrections
  FROM public.kg_classification_log orig
  JOIN public.kg_classification_log corr ON corr.corrected_from = orig.id
  WHERE orig.stage = 4
    AND corr.created_at > NOW() - INTERVAL '90 days'
  GROUP BY orig.entity_id
)
SELECT
  stc.source_type,
  COUNT(DISTINCT stc.entity_id)                                          AS total_classified,
  COUNT(DISTINCT cc.entity_id)                                           AS entities_with_corrections,
  ROUND(
    COUNT(DISTINCT cc.entity_id) * 100.0 / NULLIF(COUNT(DISTINCT stc.entity_id), 0),
    2
  )                                                                      AS correction_rate_pct,
  ROUND(
    (1 - COUNT(DISTINCT cc.entity_id) * 1.0
       / NULLIF(COUNT(DISTINCT stc.entity_id), 0)) * 100,
    2
  )                                                                      AS estimated_accuracy_pct
FROM source_type_classifications stc
LEFT JOIN correction_counts cc ON cc.entity_id = stc.entity_id
GROUP BY stc.source_type
ORDER BY correction_rate_pct DESC;
```

**Alert threshold**: Any source type with `correction_rate_pct > 10%` in the trailing 90-day window. Board minutes are expected to start higher — initial acceptable threshold is 15% for `FL Board Documents`, tightening to 10% after 6 months of HITL feedback.

**Dashboard display**: Horizontal bar chart sorted by correction rate descending. Each bar colored by severity: red > 10%, yellow 5–10%, green < 5%.

#### Metric 20: Re-Classification Volume by Trigger Type

**Purpose**: Tracks how many entities are re-classified per time period, broken down by the trigger that initiated the re-classification. A healthy system has low ongoing re-classification volume (mostly taxonomy gap resolutions). High volume from `systematic_error` triggers indicates a pipeline regression that needs investigation.

**Calculation SQL**:
```sql
SELECT
  DATE_TRUNC('week', rcj.started_at)    AS week,
  rcj.trigger_type,
  rcj.jurisdiction,
  COUNT(DISTINCT rcj.id)                AS job_count,
  SUM(rcj.entities_total)               AS entities_queued,
  SUM(rcj.entities_completed)           AS entities_completed,
  SUM(rcj.entities_failed)              AS entities_failed,
  ROUND(
    SUM(rcj.entities_completed) * 100.0
      / NULLIF(SUM(rcj.entities_total), 0),
    1
  )                                     AS completion_rate_pct,
  SUM(rcj.api_cost_usd)                 AS total_api_cost_usd
FROM public.reclassification_jobs rcj
WHERE rcj.started_at > NOW() - INTERVAL '12 weeks'
GROUP BY 1, 2, 3
ORDER BY 1 DESC, 4 DESC;
```

(The `reclassification_jobs` table is defined in Section 3 below.)

**Alert threshold**: More than 5,000 entities queued in a single week from a `systematic_error` trigger (indicates a significant pipeline regression). More than 3 jobs in `failed` status in any 7-day window.

**Dashboard display**: Stacked bar chart by week, bars colored by `trigger_type`. Secondary table with job-level detail for the current week.

## Section 3: Bulk Re-Classification Pipeline

### 3A. Scope Determination Algorithm

For each trigger type, the SQL query that identifies affected entities is provided below. All queries are production-ready against Cedar's schema.

#### Trigger 1: Domain Split

When an existing domain code is split into two child codes (per P3-S6's split migration protocol), re-classify all entities that currently hold the deprecated parent code.

```sql
-- Scope query: entities affected by a domain split
-- Parameters: :deprecated_parent_code, :split_detection_cutoff (date, typically today)
-- Volume threshold from P3-S6: if COUNT(*) < 500, scope = 'all'; else scope = 'last_12_months'

SELECT
  ked.entity_id,
  ked.domain_code           AS current_domain,
  ked.confidence            AS current_confidence,
  ke.source_slug,
  ke.jurisdiction,
  ke.created_at             AS entity_created_at,
  -- Flag entities where a prior human correction exists (will be handled in 3C conflict resolution)
  EXISTS (
    SELECT 1 FROM public.review_actions ra
    WHERE ra.entity_id = ked.entity_id
      AND ra.action IN ('approved', 'corrected')
  )                         AS has_human_verification,
  kcl.id                    AS last_classification_log_id,
  kcl.ai_reasoning          AS prior_reasoning   -- reused in re-classification prompt
FROM public.kg_entity_domains ked
JOIN public.kg_entities ke ON ke.id = ked.entity_id
LEFT JOIN public.kg_classification_log kcl
  ON kcl.entity_id = ked.entity_id
  AND kcl.stage = 4
  AND kcl.created_at = (
    SELECT MAX(kcl2.created_at) FROM public.kg_classification_log kcl2
    WHERE kcl2.entity_id = ked.entity_id AND kcl2.stage = 4
  )
WHERE ked.domain_code = :deprecated_parent_code
  -- Scope filter: apply date filter only when entity count >= 500
  -- (Conditional logic applied at application layer after COUNT check)
ORDER BY ke.created_at DESC;
```

#### Trigger 2: New Domain Added

Re-classify all entities that are currently gap-flagged in the same L2 parent as the new domain, plus low-confidence entities in the parent domain.

```sql
-- Scope query: entities affected by a new domain being added
-- Parameters: :new_domain_code, :parent_l2_code

SELECT
  ked.entity_id,
  ked.domain_code           AS current_domain,
  ked.confidence            AS current_confidence,
  tge.gap_signal_type,
  tge.gap_reasoning,        -- inject as additional context in Stage 4 prompt
  ke.source_slug,
  ke.jurisdiction,
  EXISTS (
    SELECT 1 FROM public.review_actions ra
    WHERE ra.entity_id = ked.entity_id AND ra.action IN ('approved', 'corrected')
  )                         AS has_human_verification
FROM public.kg_entity_domains ked
JOIN public.kg_entities ke ON ke.id = ked.entity_id
-- Condition A: gap-flagged entities in the same L2 parent
LEFT JOIN public.taxonomy_gap_events tge
  ON tge.entity_id = ked.entity_id
  AND tge.resolved = FALSE
  AND tge.associated_l2_domain = :parent_l2_code
WHERE (
  -- Gap-flagged in the L2 parent
  (tge.id IS NOT NULL)
  OR
  -- Low-confidence entities currently assigned to the parent L2 or its existing children
  (
    ked.domain_code LIKE (:parent_l2_code || '%')
    AND ked.confidence < 0.75
    AND NOT EXISTS (
      SELECT 1 FROM public.review_actions ra
      WHERE ra.entity_id = ked.entity_id AND ra.action IN ('approved', 'corrected')
    )
  )
)
ORDER BY ked.confidence ASC;  -- prioritize lowest-confidence first
```

#### Trigger 3: Classification Rule Updated

Re-classify entities that were originally classified by the updated rule. Requires that the `kg_classification_log` tracks which classification rule drove the assignment (via the `rule_id` column expected in the P2-S1 schema).

```sql
-- Scope query: entities classified by a specific rule that has been updated
-- Parameters: :updated_rule_id

SELECT DISTINCT
  kcl.entity_id,
  kcl.stage                 AS classified_by_stage,
  kcl.created_at            AS original_classification_at,
  ke.source_slug,
  ke.jurisdiction,
  EXISTS (
    SELECT 1 FROM public.review_actions ra
    WHERE ra.entity_id = kcl.entity_id AND ra.action IN ('approved', 'corrected')
  )                         AS has_human_verification
FROM public.kg_classification_log kcl
JOIN public.kg_entities ke ON ke.id = kcl.entity_id
WHERE kcl.rule_id = :updated_rule_id
  -- Only entities where the rule was the determining factor
  -- (stage 1, 2, or 3a — rule-based stages; not stage 4 which is AI-driven)
  AND kcl.stage IN (1, 2, 3)
ORDER BY kcl.created_at DESC;
```

#### Trigger 5: State Onboarding (Initial Classification)

When a new state is onboarded, all entities from the new state's sources are classified for the first time. This is the full initial corpus classification, not a re-classification.

### 3B. Re-Classification Pipeline — Stage Selection Matrix

Bulk re-classification does not always require the full 5-stage pipeline. The trigger type determines which stages are relevant.

| Trigger Type | Stage 1 (Structural) | Stage 2 (Metadata) | Stage 3a (Keywords) | Stage 4 (AI) | Stage 5 (Irrelevance) | Rationale |
|---|---|---|---|---|---|---|
| Domain split | No | No | No | Yes | No | Structural and metadata signals have not changed; only the taxonomy has changed. AI re-run with updated taxonomy context. |
| New domain added | No | No | No | Yes | No | Same as split: pipeline signals unchanged, taxonomy context updated. Gap-flagged entities have prior `gap_reasoning` injected as extra context. |
| Classification rule updated (Stage 1 rule) | Yes | No | No | If Stage 1 below threshold | No | Re-run only the updated stage. Stage 4 only if new Stage 1 result is below classification threshold. |
| Classification rule updated (Stage 2 rule) | No | Yes | No | If Stage 2 below threshold | No | Same pattern for Stage 2. |
| Classification rule updated (Stage 3a rule) | No | No | Yes | If Stage 3a below threshold | No | Same pattern for Stage 3a. |
| Systematic error (domain misclassification) | No | No | No | Yes | No | Domain-level errors are in the AI reasoning, not the structural rules. Full AI re-run required. |
| Systematic error (authority level misclassification) | No | Yes | No | No | No | Authority level is assigned in Stage 2 (metadata disambiguation). Re-run Stage 2 with corrected rules. |
| Systematic error (full pipeline regression) | Yes | Yes | Yes | Yes | Yes | Full re-run when the error source is unknown or spans multiple stages. |
| State onboarding | Yes | Yes | Yes | Yes | Yes | First-time classification; full pipeline required. |
| Merge conflict resolution | No | No | No | Yes | No | P3-S6 merge protocol: only entities with confidence disagreement ≥ 0.15 are re-classified. AI-only. |

### 3C. Conflict Resolution: Human-Verified Classifications

#### Detecting Human-Verified Classifications

A classification is human-verified if either of the following is true:
1. A row exists in `review_actions` for the entity with `action IN ('approved', 'corrected')`.
2. A row exists in `kg_classification_log` for the entity with `authority_level_source = 'hitl_correction'` OR `corrected_from IS NOT NULL` (indicating a human correction was applied).

```sql
-- Query: detect human-verified status before re-classifying
-- Returns TRUE if the entity has any human verification signal

SELECT EXISTS (
  SELECT 1 FROM public.review_actions ra
  WHERE ra.entity_id = :entity_id
    AND ra.action IN ('approved', 'corrected')
  UNION ALL
  SELECT 1 FROM public.kg_classification_log kcl
  WHERE kcl.entity_id = :entity_id
    AND (kcl.authority_level_source = 'hitl_correction'
         OR kcl.corrected_from IS NOT NULL)
) AS is_human_verified;
```

#### Conflict Resolution Logic

When a bulk re-classification produces a different result than the original classification:

```
CONFLICT RESOLUTION RULES
═══════════════════════════════════════════════════════════════════

Case 1: Original classification is NOT human-verified.
  → Newer classification wins automatically.
  → Original classification log row retained (append-only).
  → New classification log row inserted with corrected_from = original log id.
  → No operator action required.

Case 2: Original classification IS human-verified, AND new classification
        is IDENTICAL to the human-verified classification.
  → No conflict. Newer classification confirms human judgment. No operator action.

Case 3: Original classification IS human-verified, AND new classification
        DIFFERS from the human-verified classification.
  → Hold: do NOT auto-apply the new classification.
  → Insert row into reclassification_conflicts table (see DDL below).
  → Emit event: 'reclassification/conflict.human_verified'
  → Queue in HITL with review_priority = 'high' and queue_reason = 'human_verified_conflict'.
  → UI prompt: "Re-classification job [job_id] produced a different result than your
    previous review of this entity. The re-classification suggests [new_domain] (confidence:
    [score]). Your prior review confirmed [original_domain]. Please confirm which
    classification to apply."
  → Reviewer choices: (a) accept new classification, (b) keep human-verified, (c) defer.

Case 4: Original classification IS human-verified, AND re-classification
        is triggered by a SYSTEMATIC ERROR correction (trigger_type = 'systematic_error')
        for a pattern that MATCHES the entity.
  → Special case: even human-verified classifications may be wrong if the error
    affected the reviewer's context at time of review.
  → Treat the same as Case 3: route to HITL, but add note:
    "This entity was affected by systematic error [error_description]. The human
    verification predates the error discovery and may reflect the same error."
  → Reviewer must re-confirm even previously confirmed classifications.
═══════════════════════════════════════════════════════════════════
```

### 3D. Rate Limiting and Resource Management

#### 3D-1. Batch Sizes

| Operation | Batch Size | Rationale |
|---|---|---|
| Database reads (fetch entities + current classifications) | 500 per query page | Postgres cursor pagination, keeps memory footprint bounded |
| Stage 4 AI calls | 3 per API call (unchanged from P2-S4) | Optimal batch size per P2-S4 analysis: 50% cost reduction vs. N=1, quality maintained |
| Database writes (classification log + entity_domains update) | 100 per transaction | Balances write throughput with transaction rollback cost on failure |
| Fan-out: Inngest events dispatched per `sendMany()` call | 50 entities per dispatch | Respects Inngest's recommended batch send limit |

#### 3D-2. API Rate Limiting

Claude API rate limits (as of P2-S4 specification):
- Haiku: 4,000 RPM (requests per minute) on standard tier
- Re-classification concurrency: limited to 5 concurrent Stage 4 AI calls (lower than live classification's 50 concurrency limit)
- At batch size 3 and concurrency 5: maximum 15 entities/minute through AI = 900/hour
- For a 10K entity job: approximately 11 hours at this rate
- Operators must be informed of expected completion time when initiating large jobs

Rate limit enforcement in Inngest:

```typescript
// In the reclassify-batch step function:
// Inngest built-in rate limiting via concurrency settings (see function signature in 3D-4)
// Plus exponential backoff on 429 responses (same as P2-S4's callClassificationAPI handler)

const RECLASSIFY_RATE_LIMIT = {
  concurrency: 5,         // max parallel Stage 4 API calls
  batchSize: 3,           // entities per API call
  retryDelayMs: [2000, 4000, 8000],  // exponential backoff on rate limit
  pauseBetweenBatches: 200  // ms between batch dispatches to avoid burst
};
```

#### 3D-3. Database Write Batching

All re-classification writes are batched and transactional:
1. Read: fetch 500 entities with current classifications
2. Classify: run Stage 4 on batches of 3 until all 500 are classified
3. Write (single transaction per 100 entities):
   - `INSERT INTO kg_classification_log (...)` — new classification log entry per entity
   - `UPDATE kg_entity_domains SET domain_code = new_code, confidence = new_confidence WHERE entity_id = ...` — update current domain assignment
   - `INSERT INTO reclassification_jobs_progress (...)` — progress tracking row

If the transaction fails: the batch is retried. Already-written entities are idempotently re-written (upsert on entity_id + job_id).

#### 3D-5. Inngest Architecture

**Recommendation: Fan-out with a coordinator function and per-batch worker functions.**

A single long-running function would time out on large jobs (Inngest's default max function duration is 2 hours on Hobby plans). The fan-out pattern uses one coordinator that dispatches 50-entity batches to individual worker invocations. Each worker invocation is independent: if it fails, only its 50 entities need to be retried.

```typescript
// COORDINATOR FUNCTION
// Triggered by: admin action ("start re-classification job for trigger X")
// Responsibility: determine scope, create job record, fan out to workers

export const startReclassificationJob = inngest.createFunction(
  {
    id: "reclassification-job-start",
    concurrency: [{ limit: 2 }],  // max 2 simultaneous reclassification jobs
    retries: 1,
    timeouts: { finish: "15m" },  // coordinator should complete quickly
  },
  { event: "reclassification/job.start" },
  async ({ event, step }) => {
    const {
      trigger_type,
      trigger_detail,
      stages_to_run,
      jurisdiction,
      initiated_by
    } = event.data;

    // Step 1: Determine scope
    const entityIds: string[] = await step.run("determine-scope", async () => {
      // Execute the appropriate scope query from Section 3A
      return await runScopeQuery(trigger_type, trigger_detail, jurisdiction);
    });

    // Step 2: Create job record
    const jobId: string = await step.run("create-job-record", async () => {
      const { data } = await supabase
        .from("reclassification_jobs")
        .insert({
          trigger_type,
          trigger_detail,
          stages_to_run,
          jurisdiction,
          entities_total: entityIds.length,
          entities_queued: entityIds.length,
          status: "running",
          taxonomy_version_at_start: await getCurrentTaxonomyVersion(),
          started_by: initiated_by,
          started_at: new Date().toISOString(),
        })
        .select("id")
        .single();
      return data.id;
    });

    // Step 3: Insert per-entity tracking rows
    await step.run("create-entity-tracking", async () => {
      const trackingRows = entityIds.map(entity_id => ({
        job_id: jobId,
        entity_id,
        status: "queued",
      }));
      // Batch insert in chunks of 500
      for (let i = 0; i < trackingRows.length; i += 500) {
        await supabase
          .from("reclassification_job_entities")
          .insert(trackingRows.slice(i, i + 500));
      }
    });

    // Step 4: Fan out to worker functions (50 entities per invocation)
    const batches: string[][] = [];
    for (let i = 0; i < entityIds.length; i += 50) {
      batches.push(entityIds.slice(i, i + 50));
    }

    await step.run("fan-out-workers", async () => {
      await inngest.sendMany(
        batches.map((batch, idx) => ({
          name: "reclassification/batch.process",
          data: {
            job_id: jobId,
            batch_index: idx,
            entity_ids: batch,
            stages_to_run,
            jurisdiction,
            trigger_type,
          },
        }))
      );
    });

    return { job_id: jobId, total_entities: entityIds.length, batches_dispatched: batches.length };
  }
);

// WORKER FUNCTION
// Triggered by: coordinator fan-out
// Responsibility: classify one 50-entity batch, write results, update job progress

export const processReclassificationBatch = inngest.createFunction(
  {
    id: "reclassification-batch-process",
    concurrency: [{ limit: 5 }],  // max 5 concurrent workers (rate limiting)
    retries: 3,
    timeouts: { finish: "30m" },  // one batch of 50 at batch-size-3 = ~17 AI calls
  },
  { event: "reclassification/batch.process" },
  async ({ event, step }) => {
    const { job_id, batch_index, entity_ids, stages_to_run, jurisdiction, trigger_type } = event.data;

    for (const entity_id of entity_ids) {
      await step.run(`classify-entity-${entity_id}`, async () => {
        // 1. Fetch entity + current classification
        const entity = await fetchEntityWithCurrentClassification(entity_id);

        // 2. Check human-verified status (conflict resolution case 2/3/4)
        const isHumanVerified = await checkHumanVerifiedStatus(entity_id);

        // 3. Run the appropriate pipeline stages
        const newClassification = await runPipelineStages(
          entity,
          stages_to_run,
          jurisdiction,
          { prior_reasoning: entity.current_ai_reasoning }  // inject for domain splits
        );

        // 4. Handle conflict resolution
        if (isHumanVerified && newClassification.primary_domain !== entity.current_domain) {
          // Case 3 or 4: hold and route to HITL
          await createReclassificationConflict(job_id, entity_id, entity, newClassification);
          await updateJobEntityStatus(job_id, entity_id, "conflict");
        } else {
          // Cases 1 and 2: auto-apply
          await applyNewClassification(job_id, entity_id, entity, newClassification);
          await updateJobEntityStatus(job_id, entity_id, "completed");
        }

        // 5. Update job-level counters
        await incrementJobCounter(job_id, isHumanVerified ? "entities_skipped" : "entities_completed");

        // 6. Track API cost
        if (newClassification.api_cost_usd) {
          await incrementJobCost(job_id, newClassification.api_cost_usd);
        }
      });
    }

    return { job_id, batch_index, processed: entity_ids.length };
  }
);
```

### 3E. Rollback

#### Rollback Capability

Every bulk re-classification job is rollback-capable because:
- The original classification log row is never deleted (append-only).
- The `reclassification_job_entities` table links each entity to both its `prior_classification_log_id` and its `new_classification_log_id`.
- The `kg_entity_domains` table was updated by the job — rollback restores the prior domain_code and confidence values from the original classification log.

#### Rollback Decision Criteria

An operator should consider rollback when any of the following conditions are met within 48 hours of job completion:

| Signal | Threshold | Action |
|---|---|---|
| Correction rate on re-classified entities (new corrections arriving faster than pre-job baseline) | > 2× the pre-job correction rate in the same domain | Investigate; consider rollback |
| HITL queue depth spike immediately post-job | > 200 entities queued attributable to re-classified batch | Strong rollback indicator |
| User-reported classification errors from practice owners in re-classified domains | Any user report within 72 hours of job | Manual review + possible rollback |
| Average confidence of new classifications is LOWER than original classifications | Mean Δconfidence < −0.10 | Rollback: the job made classifications worse on average |
| Confusion pattern detector fires a new pattern involving domains touched by the job | New confusion pair emerges within 7 days of job completion | Investigate; rollback if confirmed |

The `taxonomy_version_at_start` column on `reclassification_jobs` is critical: it records the taxonomy version that was active when the job ran. If the rollback occurs after a subsequent taxonomy change, the operator must account for the fact that rolling back may re-introduce entities with domain codes that have since been deprecated.

#### Rollback SQL

```sql
-- Rollback a reclassification job: restore the prior classification for all completed entities.
-- Parameters: :job_id
-- Execute in a transaction. Review output of the SELECT before committing.

BEGIN;

-- Step 1: Preview what will be restored (run as SELECT first, then convert to UPDATE/INSERT)
SELECT
  rje.entity_id,
  kcl_prior.domain_code                             AS prior_domain,
  kcl_prior.confidence                              AS prior_confidence,
  kcl_new.domain_code                               AS current_domain_to_undo,
  kcl_new.confidence                                AS current_confidence_to_undo
FROM public.reclassification_job_entities rje
JOIN public.kg_classification_log kcl_prior
  ON kcl_prior.id = rje.prior_classification_log_id
JOIN public.kg_classification_log kcl_new
  ON kcl_new.id = rje.new_classification_log_id
WHERE rje.job_id = :job_id
  AND rje.status = 'completed';

-- Step 2: Restore kg_entity_domains to prior values
UPDATE public.kg_entity_domains ked
SET
  domain_code  = kcl_prior.domain_code,
  confidence   = kcl_prior.confidence
FROM public.reclassification_job_entities rje
JOIN public.kg_classification_log kcl_prior
  ON kcl_prior.id = rje.prior_classification_log_id
WHERE rje.job_id = :job_id
  AND rje.status = 'completed'
  AND ked.entity_id = rje.entity_id
  AND ked.is_primary = TRUE;

-- Step 3: Insert rollback audit entries (one per entity, marking the restoration)
INSERT INTO public.kg_classification_log (
  entity_id,
  stage,
  domain_code,
  confidence,
  authority_level_source,
  jurisdiction,
  corrected_from,
  ai_reasoning
)
SELECT
  rje.entity_id,
  0                                                   AS stage,  -- stage 0 = system rollback
  kcl_prior.domain_code,
  kcl_prior.confidence,
  'lifecycle_transition'                              AS authority_level_source,
  rcj.jurisdiction,
  kcl_new.id                                          AS corrected_from,
  jsonb_build_object(
    'rollback_job_id', :job_id,
    'rollback_reason', :rollback_reason,
    'chain_of_thought', 'System rollback: restoring classification prior to reclassification job'
  )                                                   AS ai_reasoning
FROM public.reclassification_job_entities rje
JOIN public.reclassification_jobs rcj ON rcj.id = rje.job_id
JOIN public.kg_classification_log kcl_prior ON kcl_prior.id = rje.prior_classification_log_id
JOIN public.kg_classification_log kcl_new   ON kcl_new.id   = rje.new_classification_log_id
WHERE rje.job_id = :job_id
  AND rje.status = 'completed';

-- Step 4: Update job status to rolled_back
UPDATE public.reclassification_jobs
SET
  status          = 'rolled_back',
  rollback_reason = :rollback_reason,
  rollback_at     = NOW()
WHERE id = :job_id;

COMMIT;
```

### 4C. Partitioning Strategy

At the projected volume of ~900K rows across 3 years, **no partitioning is required**. Postgres handles tables up to tens of millions of rows efficiently with appropriate indexes.

If Cedar expands to all 50 states and/or if re-classification jobs run more frequently than anticipated, the following partitioning approach is recommended at ~5M rows:

- **`kg_classification_log`**: Partition by `created_at` (monthly range partitions). The most common query patterns filter by `created_at > NOW() - INTERVAL X`, so date partitioning provides direct partition pruning.
- **`authority_level_transitions`**: Partition by `jurisdiction` if Cedar expands to 10+ states. Jurisdiction-scoped queries are the primary access pattern.
- **`taxonomy_gap_events`**: No partitioning needed even at scale — the table is expected to shrink as the taxonomy matures, not grow proportionally.

### 4D. Archival Approach

Cedar's audit trail is legally significant — it documents when a practice was notified of a regulatory change and what classification was applied. Archival must preserve queryability.

**Archival policy**:
- **Hot tier** (Supabase primary): All rows from the last **24 months**. Fully indexed and queryable.
- **Warm tier** (Supabase or S3 via pg_partman): Rows older than 24 months, retained for **7 years** total. Queryable via a cold query path (acceptable latency 5–30 seconds). Required for compliance audit requests.
- **Delete**: No deletion. Cedar's append-only audit principle extends to archival — records are moved to warm/cold storage but never deleted.

**Implementation**: At the 24-month boundary, a monthly Inngest cron function moves `kg_classification_log` rows older than 24 months to a `kg_classification_log_archive` table (identical schema). Queries involving historical data must union across both tables. A `classification_log_full` view unifies them transparently for reporting queries.

```sql
-- Archival view (transparent union across hot and warm storage)
CREATE OR REPLACE VIEW public.classification_log_full AS
  SELECT * FROM public.kg_classification_log
  UNION ALL
  SELECT * FROM public.kg_classification_log_archive;
```
