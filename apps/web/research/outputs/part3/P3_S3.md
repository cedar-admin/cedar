## Preamble: How CMS and Professional Content Enters the Pipeline

CMS transmittals, NCDs, LCDs, MLN articles, and professional guidelines are **non-FR, non-openFDA sources**. They do not carry CFR references as primary metadata (though they may cite CFR provisions in body text). The pipeline therefore cannot resolve them through Stage 1 (Structural/CFR). They enter at **Stage 2 (Metadata: A + D + B)** and proceed through **Stage 3a (Keywords)** unless the early-resolution check (A+D+B ≥ 0.60) fires first.

For CMS-published content, the source agency is always CMS — a **High-relevance agency (A = 0.45)** per P1-S3. This means A+D alone can reach 0.60 (0.45 + 0.15 = 0.60) for the highest-authority CMS document types, triggering early resolution and bypassing keyword scoring entirely for the most significant transmittals and NCDs.

For professional association content, A-scores are assigned based on the proportion of output relevant to Cedar's scope. These organizations are not in the P1-S3 agency tier map (which covers federal FR/openFDA agencies) — they require their own A-score assignments in the range 0.05–0.45, calibrated to reflect how consistently relevant their output is to Cedar's 14 practice types.

- **Medicare Program Integrity Manual (MPIM)** — Publication 100-08; defines fraud/audit/compliance requirements

### 1C. Transmittal Type D-Score Assignments

CMS transmittals are not Federal Register documents, so P1-S3's FR document type scoring does not apply directly. The following D-scores adapt the FR document type framework to the CMS transmittal authority structure:

| Transmittal Type | Description | D-Score | Rationale |
|---|---|---|---|
| **New Policy** | Transmittal establishes a new coverage, billing, or compliance requirement | **0.15** | Equivalent to Final Rule — creates binding new obligations |
| **Revised Policy** | Transmittal changes an existing policy in a material way | **0.13** | High authority; comparable to final rule correction/amendment |
| **Annual Fee Schedule Update** | Recurring transmittal updating RVUs, conversion factor, or code values | **0.12** | High operational impact; predictable but always actionable |
| **Code Addition/Deletion** | Transmittal adding, revising, or deleting CPT/HCPCS codes | **0.12** | Directly affects billing; practice impact is immediate |
| **Clarification/Correction** | Transmittal clarifying ambiguous policy or correcting error | **0.10** | May change compliance behavior; lower urgency |
| **Administrative/Process Update** | Transmittal updating forms, process timelines, or contractor instructions | **0.06** | Low practice-facing impact; often MAC-internal |
| **Educational/Informational** | Transmittal sharing information without changing policy | **0.04** | Similar to MLN articles; no regulatory force |

**Implementation note:** Cedar should extract the transmittal's `CR Type` field (Change Request type, a structured field in CMS's transmittal format) to determine transmittal type. When `CR Type` is not available, the pipeline should use keyword classification on the transmittal title: presence of "REVISED," "NEW," "CLARIFICATION," or "CORRECTION" to assign D-score tier.

### 1D. ICD/CPT Code Signal Approach

When a transmittal body references specific ICD-10 or CPT codes, those codes signal the clinical domain. Cedar faces a build/buy/defer decision on CPT-to-domain mapping:

**Option A: Full CPT code range → Cedar domain lookup table**

CPT code ranges map cleanly to clinical specialties. A static lookup covering the major ranges relevant to Cedar's 14 practice types would resolve ~80% of domain assignments deterministically:

| CPT Range | Clinical Area | Cedar Domain |
|---|---|---|
| 10000–19999 | Skin, subcutaneous, and accessory structures; breast | `clinical-operations` → `fda-regulation.devices` (for devices) |
| 20000–29999 | Musculoskeletal | `clinical-operations` (Pain Management, Chiropractic, Regenerative Medicine) |
| 36000–36999 | Hemic and lymphatic; IV/infusion | `clinical-operations` (IV Therapy, Weight Management) |
| 50000–59999 | Urinary; reproductive | `clinical-operations` (HRT, ACOG-covered services) |
| 60000–69999 | Endocrine, nervous system | `clinical-operations` (Functional Medicine, Pain Management) |
| 70000–79999 | Radiology | `clinical-operations` (diagnostic imaging) |
| 80000–89999 | Pathology/laboratory | `clinical-operations.diagnostics` (Functional Medicine, Primary Care) |
| 90000–99999 | E/M, medicine | `medicare-billing.physician-services` (ALL billing practices) |
| 99202–99215 | Office E/M visits | `medicare-billing.physician-services` (highest relevance) |
| 99421–99423 | Online digital E/M | `telehealth.reimbursement` |
| 99441–99443 | Telephone E/M | `telehealth.reimbursement` |
| 0001T–9999T | Category III (emerging technology) | `fda-regulation.devices` or `clinical-operations` |

**Option B: Keyword signal only** — Use CPT code references as keyword signals (Strong signal if a recognized code for a Cedar-relevant service is cited) without maintaining a full lookup table.

**Option C: AI classification for CPT signals** — Pass transmittals with CPT references to Stage 4 (AI) for domain resolution.

**Recommendation: Hybrid approach (A+B)**

Build a targeted CPT range lookup covering the ~200 specific codes most relevant to Cedar's 14 practice types (not all 10,000+ CPT codes). These are the codes that appear in NCD/LCD coverage conditions and in MPFS transmittals for Cedar target services. Use regex pattern matching against CPT code mentions in transmittal text. When a recognized code is found, apply a **+0.03 bonus** (a new B modifier for CMS-specific rules) and add the corresponding clinical domain as a secondary cross-classification. When an unrecognized code is found, treat it as a Moderate keyword signal (K = 0.15).

**Cost/benefit:** Building a full 10,000-code CPT lookup provides marginal improvement over a targeted 200-code lookup. The 200-code set can be maintained in a single lookup table (`cms_code_domains`) and updated annually when CPT publishes. AI classification should be reserved for transmittals where the CPT codes fall outside the lookup table — a much smaller set requiring AI handling.

## Section 2: National Coverage Determinations (NCDs) & Local Coverage Determinations (LCDs)

### 2A. NCD Cross-Classification Approach

Each NCD covers a specific medical service or procedure and contains two functionally distinct layers:

1. **Clinical layer:** Indications, contraindications, documentation requirements — what makes the service medically necessary
2. **Billing/coverage layer:** Medicare coverage conditions, coding, and limitations — when and how CMS will pay

**Primary domain assignment:** `medicare-billing.coverage-determinations`

This is the correct primary domain for all NCDs. The NCD is fundamentally a coverage decision — it defines when Medicare will and will not pay. The clinical criteria within an NCD are subordinate to the coverage determination purpose.

**Cross-classification into clinical domain:** Every NCD should receive a secondary classification into the most specific available clinical domain node from P1-S4. The clinical cross-classification is assigned based on the service/procedure category:

| NCD Service Category | Secondary Clinical Domain |
|---|---|
| Diagnostic imaging | `clinical-operations` |
| Laboratory testing | `clinical-operations.diagnostics` |
| Chiropractic manipulation | `clinical-operations` (Chiropractic) |
| Pain management procedures | `clinical-operations` (Pain Management) |
| Telehealth/virtual services | `telehealth.reimbursement` |
| Weight management | `clinical-operations` (Weight Management) |
| Cardiovascular services | `clinical-operations` |
| Psychiatric/behavioral health | `clinical-operations` (Integrative Medicine, Functional Medicine) |
| Durable medical equipment | `medicare-billing.coverage-determinations` (no clinical cross-classification needed) |
| Preventive services | `clinical-operations` (Primary Care) |
| Regenerative/cellular therapy | `fda-regulation.biologics` (secondary); `clinical-operations` (Regenerative Medicine) |
| Compounded drugs | `compounding.503a` or `compounding.503b` |
| Devices | `fda-regulation.devices` |

**Classification rule:** An NCD receives ONE primary domain code (`medicare-billing.coverage-determinations`) and one or more secondary domain codes from the table above. Both primary and secondary domains are stored in `secondary_domain_codes[]` in the `classification_rules` record. The NCD entity's `DomainAssignment[]` should have `is_primary = true` for `medicare-billing.coverage-determinations` and `is_primary = false` for all clinical cross-classifications.

### 2B. LCD Geographic Scope and MAC Tagging

**The problem:** LCDs are MAC-specific. Florida has two Medicare Administrative Contractors:
- **Palmetto GBA** — Part A, Home Health, Hospice (MAC Jurisdiction J/M overlap; Palmetto handles FL Part A)
- **First Coast Service Options (FCSO)** — Part B (Medicare Jurisdiction N, covering FL + PR + USVI)

LCDs from Palmetto GBA and FCSO are relevant only to practices billing Medicare in Florida. LCDs from other MACs (e.g., Novitas, WPS, CGS) are irrelevant to Cedar's current FL-first scope unless the LCD covers a service also relevant to FL practices and may become an NCD.

**Jurisdiction schema decision (aligning with P2-S1):**

P2-S1 defines `jurisdiction` as a string field on classification rules. The existing schema uses values like `'federal'` and `'FL'`. LCDs introduce a sub-state jurisdiction level.

**Recommended approach: `jurisdiction = 'FL'` with a separate `mac` field in `rule_config` JSONB**

Rationale:
1. P2-S1's `RuleSet` interface loads rules by `jurisdiction` string. Adding `FL-MAC-PALMETTO` as a jurisdiction value would require Cedar to load and manage separate rule sets for each MAC — creating combinatorial complexity as more states are added.
2. The `rule_config` JSONB column in `classification_rules` already accommodates source-specific metadata. Adding `mac_id` and `mac_name` to `rule_config` keeps the jurisdiction system clean while preserving MAC specificity.
3. At query time, Cedar can filter `WHERE rule_config->>'mac_id' IN ('palmetto-gba', 'first-coast')` when loading FL LCD rules, without touching the jurisdiction field.

**MAC field schema for LCD rule_config:**

```json
{
  "content_type": "lcd",
  "mac_id": "first-coast",
  "mac_name": "First Coast Service Options",
  "mac_jurisdiction": "J-N",
  "mac_part": "B",
  "lcd_id": "L12345",
  "effective_date": "2024-01-01"
}
```

**Florida MAC coverage matrix:**

| MAC | Contractor | Part | Jurisdictions Served | Relevant for Cedar FL? |
|---|---|---|---|---|
| Palmetto GBA | Palmetto Government Benefits Administrators | A, HH, Hospice | AL, GA, TN, VA, WV, KY, NC, SC, FL | Yes — FL Part A |
| First Coast Service Options | First Coast Service Options, Inc. | B | FL, PR, USVI (Jurisdiction N) | Yes — FL Part B |

**Implementation:** Cedar's LCD ingestion pipeline should filter for Palmetto GBA and First Coast LCDs only. When new LCDs are published to the CMS LCD database, the ingestion job checks `contract_number` against the FL MAC identifiers before queuing for classification.

### 2C. Practice-Type Relevance Scoring for NCDs/LCDs

**Core challenge:** An NCD about chiropractic manipulation spinal manipulation therapy (SMT) is High relevance for Chiropractic practices and None relevance for Med Spas. Practice-type relevance cannot be determined from the composite score alone — it requires service-level matching.

**Approach: Title + covered service → practice-type relevance mapping**

The NCD/LCD title and the service it covers are sufficient to assign practice-type relevance without AI classification in most cases. The following signal hierarchy applies:

1. **Service code signals (highest reliability):** If the NCD/LCD specifies covered CPT/HCPCS codes, use the Cedar CPT range lookup (Section 1D) to assign practice-type relevance. CPT codes are objective and do not require text parsing.

2. **Title keyword signals (secondary):** Extract the clinical service from the NCD/LCD title. Apply the practice-type relevance mapping from P1-S4's L1 domain definitions. Example: "Acupuncture for Chronic Low Back Pain" → Pain Management (High), Integrative Medicine (High), Chiropractic (Medium), all others Low or None.

3. **Fallback mapping table:** Cedar should maintain a `ncd_practice_relevance` table mapping NCD ID numbers to practice types. This table is seeded from the NCD catalog at initial build and updated as new NCDs are issued. LCDs use the same approach but are keyed by `lcd_id + mac_id`.

**Practice-type relevance interaction with P1-S4:** The `practice_type_relevance` construct from P1-S4 is domain-level. NCD/LCD relevance is entity-level — more granular. The entity-level relevance should take precedence over the domain-level default. When an NCD is classified under `medicare-billing.coverage-determinations`, the domain-level relevance for that domain defaults to the P1-S4 table values. The NCD-specific relevance mapping overrides those defaults for alert delivery: a practice owner enrolled in "Chiropractic" as their practice type receives the SMT NCD alert; a practice owner enrolled in "Med Spa" does not.

## Section 3: Medicare Learning Network (MLN) Articles

### 3A. Authority Level

MLN articles have **no regulatory authority**. They are educational/informational documents published by CMS to help providers understand billing rules, coverage policies, and compliance requirements in plain language. They do not create new obligations, amend existing requirements, or carry the force of law. The underlying authority always resides in the transmittal, NCD, LCD, or regulation that the MLN article explains.

**Implication for Cedar:** MLN articles are supporting reference material, not primary regulatory alerts. Cedar should ingest them for two purposes:
1. As **cross-references** linking to the underlying transmittal or NCD they explain
2. As **plain-language summaries** that can enhance Cedar's AI summaries of corresponding regulatory changes (Agent 2 classification layer)

### 3B. Classification Differentiation from Transmittals and NCDs

| Attribute | CMS Transmittal | NCD | MLN Article |
|---|---|---|---|
| Authority level | Binding — updates Medicare manuals | Binding — governs coverage | None — educational only |
| Creates new obligations? | Yes (New Policy type) | Yes (coverage conditions) | No |
| D-score | 0.04–0.15 (by transmittal type) | 0.12 (coverage determination) | **0.04** |
| A-score | 0.45 (CMS = High) | 0.45 (CMS = High) | 0.45 (CMS = High) |
| Primary domain | Per manual chapter mapping | `medicare-billing.coverage-determinations` | Per topic (derived from linked source) |
| Alert delivery? | Yes — primary alert | Yes — primary alert | No — supplemental reference only |
| Cedar surfacing | Change feed, alerts | Change feed, alerts | Knowledge library only |

**D-score for MLN articles: 0.04**

This is the lowest non-zero D-score, equivalent to a presidential determination or proclamation in the FR scoring model — informational, no binding force. An MLN article from CMS still carries A = 0.45, so its composite A+D baseline is 0.49 before keywords. With keyword matches, most MLN articles will clear the 0.60 relevant threshold and be ingested — but they should be **flagged as informational, not regulatory** via a `content_authority` metadata field.

### 3C. Linking MLN Articles to NCDs/LCDs and Transmittals

MLN articles routinely reference the transmittal, NCD, or LCD they explain. Cedar should extract these references at ingestion and store them as entity-to-entity links. The linking approach:

1. **Extract MLN article number** from the document (e.g., "MLN MM12345")
2. **Extract referenced transmittal CR number** if present (e.g., "Change Request 12345")
3. **Extract referenced NCD/LCD ID** if present (e.g., "NCD 70.1," "LCD L33550")
4. **Store in `entity_references` table** (or equivalent relationship table): `{from_entity: mlnArticleId, to_entity: transmittalId, reference_type: 'explains'}`

This linking enables Cedar to surface the MLN article as a "plain language explanation" alongside the primary regulatory alert for a transmittal or NCD, without conflating the authority levels.

### 4B. D-Score by Document Type for Professional Associations

Professional association documents are not Federal Register types, but they carry differentiated authority levels within their own hierarchy. D-scores are adapted from the P1-S3 framework by authority analogy:

| Document Type | D-Score | Authority Analogy | Description |
|---|---|---|---|
| **Clinical Practice Guideline (evidence-based)** | **0.13** | Close to Final Rule | Systematic evidence review; most authoritative professional content; directly shapes clinical standard of care |
| **Practice Bulletin / Technical Report** | **0.12** | Proposed Rule level | Detailed technical guidance; strong evidence base; shapes billing documentation standards |
| **Committee Opinion / Position Statement** | **0.10** | Notice (final guidance) | Expert consensus; no regulatory force but highly cited; affects standard of care defense |
| **Clinical Recommendation / Advisory** | **0.09** | Notice (determination) | Targeted guidance on a specific clinical question; moderate authority |
| **Regulatory Alert / Member Alert** | **0.12** | Notice (coverage determination, enforcement) | High urgency; typically summarizes an external regulatory change — the underlying change is the real regulatory event |
| **CPT Code Update** (AMA) | **0.15** | Final Rule equivalent | Binding for Medicare billing — CPT codes are incorporated by reference into CMS fee schedules; highest D-score among professional content |
| **E/M Documentation Guideline** (AMA) | **0.13** | Close to Final Rule | CMS adopts AMA E/M guidelines for Medicare; effectively binding |
| **Advocacy Position Paper** | **0.06** | ANPRM / RFI level | Signals potential regulatory direction; not binding; useful for trend monitoring |
| **Educational Material / CME Content** | **0.04** | Notice (informational) | No regulatory force; informational only |
| **Press Release / News Item** | **0.02** | Proclamation level | Minimal signal; rarely actionable |
| **Journal Article / Research** | **0.04** | Notice (informational) | Evidence contributing to future guidelines; not directly actionable |
| **Legislative Testimony** | **0.08** | Notice (petition/RFI) | Signals advocacy positions; may predict regulatory change; not binding |

### 4C. Cross-Reference Enrichment for Professional Guidelines

**The problem:** Professional guidelines regularly cite external authorities — federal regulations, state statutes, USP chapters, CMS transmittals, and other professional standards. When an ACOG Practice Bulletin cites 21 CFR Part 216 (bulk drug substance lists), the bulletin should inherit `compounding.bulk-substances` as a secondary domain classification in addition to its primary clinical domain.

**Citation patterns in professional guidelines:**

Professional association documents use several distinct citation patterns that differ from federal regulatory documents:

| Source Type | Common Citation Patterns | Example |
|---|---|---|
| CFR provisions | "21 C.F.R. § 503A," "42 C.F.R. Part 410," "per 21 CFR 1306" | PCCA citing 21 CFR 503A restrictions |
| USP chapters | "USP <797>," "USP Chapter <800>," "USP <825>" | PCCA/PCAB citing sterile compounding standards |
| CMS transmittals | "Change Request 12345," "Transmittal R4321CP," "CR 8XXX" | AMA citing CMS billing change |
| NCDs/LCDs | "NCD 70.1," "LCD L33550," "Local Coverage Determination for..." | AAFP citing coverage policy |
| Federal statutes | "FDCA § 503A," "21 U.S.C. § 353b," "Social Security Act § 1848" | APC citing DQSA provisions |
| State statutes | "Florida Statutes § 456," "F.S. 465," "Fla. Admin. Code" | FMA citing FL Medical Practice Act |
| AMA CPT codes | "CPT 99213," "Current Procedural Terminology" | AAFP citing billing codes |
| Other association guidelines | "Per AHA guidelines," "ACC/AHA statement," "ADA Standards of Care" | Cross-association citation |

**Detection approach:**

The P3-S1 federal reference detection patterns (designed for FL administrative rules citing federal law) require adaptation for professional guidelines. Professional guidelines use the same CFR citation patterns but also extensively use USP chapter references, statute section citations, and NCD/LCD identifiers that P3-S1 did not need to handle.

Cedar should implement a **professional-guidelines-specific citation parser** as an extension of the P2-S1 `CitationParser` interface:

```typescript
interface ProfessionalGuidanceCitationParser extends CitationParser {
  jurisdiction: 'professional';
  parse(raw: string): ParsedCitation | null;

  // Additional patterns beyond CFR:
  patterns: {
    cfr: RegExp;              // Same as federal parser: /\d+\s+C\.?F\.?R\.?\s*(?:§|Part|§§)\s*[\d.]+/gi
    usp_chapter: RegExp;      // /USP\s*<?(\d+)>?/gi — captures USP chapter number
    ncd_lcd: RegExp;          // /(?:NCD|LCD)\s+([A-Z\d.]+)/gi
    cms_transmittal: RegExp;  // /(?:CR|Change Request|Transmittal)\s+[A-Z]?\d+/gi
    federal_statute: RegExp;  // /(?:\d+\s+U\.S\.C\.|FDCA|SSA|HIPAA|HITECH)\s*§\s*[\d\w.]+/gi
    state_statute: RegExp;    // /(?:F\.?S\.?|Fla\.?\s+Stat\.?|Florida\s+Statutes?)\s*§\s*[\d.]+/gi
    ama_cpt: RegExp;          // /CPT\s*(?:code\s*)?\d{5}/gi
  };
}
```

**Inherited classification interaction with own classification:**

When a professional guideline document cites external sources, the inherited domain classification is treated as a **secondary cross-classification**, not a primary domain override. Priority rules:

1. The guideline's own primary domain (based on its organization and document type) remains `is_primary = true`
2. Inherited domains from CFR citations are added as secondary with `assigned_by = 'keyword'` (the citation text is the keyword signal) and `is_primary = false`
3. Inherited domains from NCD/LCD citations inherit the specific coverage domain plus any practice-type relevance
4. Inherited domains from USP chapter citations map to `compounding.sterile` (USP <797>), `compounding.503a` (USP <795>), or `business-operations` (USP <800> hazardous drugs)
5. When the inherited domain is the same as the guideline's own primary domain (e.g., an ACOG bulletin citing HRT-related CFR provisions), no change — the confidence is reinforced but no duplicate entry

**Confidence assignment for inherited classifications:** Inherited domains carry a confidence of **0.75** (lower than structural classification at 1.0, higher than pure keyword at 0.50) because the citation establishes a clear connection to the referenced domain, but the guideline is discussing rather than implementing the referenced requirement.

### 7B. Stage 2 Metadata Handling for Non-FR Sources

Because CMS and professional association content does not carry FR-style `type` and `action` fields, the Stage 2 metadata function requires source-type-specific handlers:

```typescript
// Extension to stage2Metadata for non-FR sources
async function stage2MetadataForCMS(
  entity: CMSEntity,
  ruleSet: RuleSet,
  input: StageOutput
): Promise<StageOutput> {
  // A-score: CMS = High = 0.45
  input.score_components.A = 0.45;

  // D-score: derive from source_type and metadata
  const dScore = deriveCMSDocumentDScore(entity);
  input.score_components.D = dScore;

  // B-score: CPT code bonus, open comment period
  const bScore = deriveCMSBonuses(entity);
  input.score_components.B = bScore;

  return input;
}

function deriveCMSDocumentDScore(entity: CMSEntity): number {
  switch (entity.source_type) {
    case 'cms_transmittal':
      return getTransmittalDScore(entity.transmittal_type);
    case 'cms_ncd':
      return entity.is_proposed ? 0.12 : 0.15;
    case 'cms_lcd_fcso':
    case 'cms_lcd_palmetto':
      return entity.is_draft ? 0.12 : 0.15;
    case 'cms_mln':
      return 0.04;
    default:
      return 0.06;
  }
}

function getTransmittalDScore(transmittalType: string): number {
  const dScores: Record<string, number> = {
    'new_policy': 0.15,
    'revised_policy': 0.13,
    'annual_fee_schedule': 0.12,
    'code_update': 0.12,
    'clarification': 0.10,
    'correction': 0.10,
    'administrative': 0.06,
    'informational': 0.04,
  };
  return dScores[transmittalType] ?? 0.06;
}
```

### 7C. CPT Code Bonus Implementation

The B = +0.03 CPT code bonus fires in Stage 2 when a CMS transmittal or NCD/LCD body references a CPT code from Cedar's targeted code set. Implementation:

```typescript
// cms_code_domains table (seeded with ~200 targeted CPT codes)
// Schema: { cpt_code: string, cpt_range_start: number, cpt_range_end: number,
//           cedar_domain: string, practice_types: string[] }

function deriveCMSBonuses(entity: CMSEntity): number {
  let bonus = 0;

  // CPT code match bonus
  if (hasCPTCodeMatch(entity.extracted_codes, cedarCPTLookup)) {
    bonus += 0.03;
    entity.metadata.cpt_domain_boost = true;
  }

  // Open comment period bonus (NCD/LCD drafts)
  if (entity.is_open_for_comment && entity.comment_period_closes_at > new Date()) {
    bonus += 0.02;
  }

  return Math.min(bonus, 0.10); // Cap at P1-S3 maximum
}
```

### 7D. Professional Association Citation Parser

The cross-reference enrichment system for professional guidelines plugs into the existing pipeline after Stage 3a (keywords):

```typescript
// New stage: 3b - Citation enrichment (runs after Stage 3a for professional content)
async function stage3bCitationEnrichment(
  entity: ProfessionalGuidelineEntity,
  ruleSet: RuleSet,
  input: StageOutput
): Promise<StageOutput> {

  const citations = professionalGuidanceCitationParser.parse(entity.body_text);

  for (const citation of citations) {
    // CFR citations → inherited domain classification
    if (citation.type === 'cfr') {
      const inheritedDomain = lookupCFRDomain(citation.title, citation.part);
      if (inheritedDomain) {
        input.classifications.push({
          domain_code: inheritedDomain,
          confidence: 0.75,
          is_primary: false,
          assigned_by: 'keyword',
          rule_id: 'citation-enrichment-cfr'
        });
      }
    }

    // USP chapter citations → compounding subdomain
    if (citation.type === 'usp_chapter') {
      const uspDomain = lookupUSPChapterDomain(citation.chapter_number);
      if (uspDomain) {
        input.classifications.push({
          domain_code: uspDomain,
          confidence: 0.80, // USP chapters are precise mappings
          is_primary: false,
```
